# Architecture Review Board Report — ARB-001

**Date:** 2026-06-08  
**Scope:** ADR-001 through ADR-010 — Full architecture review  
**Classification:** INTERNAL — RESTRICTED  
**Review Authority:** Architecture Review Board  
**Status:** FINDINGS ISSUED — responses required before Sprint 1

---

## Executive Position

The PSL One architecture is technically coherent and the individual decisions are defensible in isolation. The stack choices (NestJS, Next.js 15, Kafka, PostgreSQL, AWS) are proven at scale and the ADRs demonstrate real options analysis.

However, **three systemic issues cut across all ADRs** and pose programme-level risk:

1. **The cost model is materially wrong.** The infrastructure cost estimates are understated by a factor of 3–5×. At the correct cost level, Year 1 infrastructure spend likely exceeds Year 1 revenue.

2. **Kafka is over-engineered for Phase 1.** A platform with an unverified user base, 15 services that don't exist yet, and a 12-week build window is being asked to implement a fully event-driven architecture with Transactional Outbox Pattern in every service. This is the right long-term direction but the wrong Phase 1 complexity level.

3. **Several decisions are one-way doors** with no documented reversal strategy. Cognito user migration, Aurora multi-cluster cost commitment, and GraphQL Federation lock-in are all accepted without adequate acknowledgement of the exit cost.

The board issues **4 BLOCKING findings** and **11 ADVISORY findings**. Blocking findings must be resolved before Sprint 1 begins.

### ADR-001: Authentication (AWS Cognito)

#### FINDING 001-A — ADVISORY: Cognito MAU cost not modelled at target scale

The ADR states "free to 50K MAU" as the key cost driver, but does not model cost at the Year 3–5 targets stated in the BRD (2M registered, 400K MAU).

**Current cost model:** Free  
**Actual cost at 400K MAU:** 350,000 MAU × $0.0055 = $1,925/month = approximately R35,000/month  
**Actual cost at 2M MAU:** 1,950,000 MAU × $0.0055 = $10,725/month = approximately R197,000/month (~R2.3M/year)

At 2M MAU, Cognito alone consumes the entirety of the Year 1 revenue target *per year* in perpetuity. The ADR dismissed Auth0 for being "~R5M/year at 400K MAU" but understated the Cognito cost at the same scale by an order of magnitude. Auth0 at 400K MAU is ~$280,000/year; Cognito is ~$420,000/year. Auth0 is actually cheaper at scale than the ADR implies.

**Required action:** Model Cognito cost at 50K, 200K, 400K, and 2M MAU. Include in the financial model before accepting this ADR.

---

#### FINDING 001-B — BLOCKING: Distributed transaction in POPIA consent flow has no rollback

The consent flow is:
```
Cognito creates user → Lambda trigger fires → Lambda calls Identity Service 
→ Identity writes ConsentRecord → Identity publishes Kafka event
```

This is a distributed transaction across three systems with no saga/compensation mechanism. The failure modes are:

- Lambda fails to call Identity Service → Cognito user exists, no ConsentRecord, no Kafka event. The fan can log in but has no loyalty account, no wallet, no fantasy access. No alert fires.
- Identity writes ConsentRecord but Kafka publish fails → Loyalty/Wallet/Fan services never create accounts. The fan exists but the platform is broken for them.
- Cognito creates user but Lambda never fires (Lambda cold start timeout) → Same as above.

For a POPIA-regulated platform, a user with a Cognito account but no ConsentRecord is in an undefined legal state: did they consent or not?

**Required action:** Define an explicit reconciliation job that detects Cognito users with missing ConsentRecords and either re-triggers the registration flow or quarantines the account. Add an alert for this condition. Document the POPIA position for accounts in this state.

---

#### FINDING 001-C — ADVISORY: Cognito migration is a one-way door with high exit cost

The ADR acknowledges migration complexity but does not quantify it. The reality: AWS does not export Cognito password hashes. A migration away from Cognito requires one of:

- A "just-in-time" migration (on login, verify against Cognito, re-hash in new system) taking 6–18 months to fully migrate the user base
- A forced password reset for all users simultaneously
- Maintaining Cognito in parallel with the new system indefinitely

At 2M users, either option is a major product incident. This exit cost should be explicitly accepted as a programme risk, not buried in "migration complexity."

**Required action:** Add a programme risk entry: "AUTH-001: Cognito exit requires forced password reset at migration." Owner: CTO. Accept or mitigate explicitly.

### ADR-002: Monorepo (Turborepo + pnpm)

#### FINDING 002-A — ADVISORY: Turbo remote cache requires Vercel account or self-hosting

The ADR references "Remote cache (Vercel Turborepo)" as a benefit, but does not specify whether the team will use Vercel's managed remote cache or self-host. Vercel's remote cache is free for open-source projects but requires a Vercel account for private repositories. The cost is part of Vercel's plan tiers.

Alternatives: Turborepo supports any HTTP remote cache endpoint. AWS S3 + a lightweight cache server is a zero-cost alternative that keeps the infrastructure within the AWS ecosystem.

**Required action:** Decide: Vercel remote cache (requires Vercel account/billing) or self-hosted Turborepo remote cache on S3. Document in ADR-002.

---

#### FINDING 002-B — ADVISORY: 15 services in one repo creates deployment coupling risk

All 15 services share `packages/event-schemas`. A breaking change to any shared package (even accidentally) will fail the build for all services simultaneously. The "affected-only" CI strategy mitigates test/build time but not the deployment coupling risk: if `packages/event-schemas` has a breaking change, every consumer must be updated in the same PR or the build breaks.

This is not a reason to use separate repos, but it is a reason to establish a strict versioning/deprecation policy for shared packages — which none of the ADRs document.

**Required action:** Define a shared package change protocol: additive-only changes without version bump, breaking changes require major version bump and migration plan. Add to CLAUDE.md or a dedicated ADR.

### ADR-003: API Strategy (GraphQL Federation)

#### FINDING 003-A — BLOCKING: Apollo Federation supergraph composition is a coordination bottleneck not addressed

The ADR requires `rover supergraph compose` in CI to validate composition before merge. This means that if any one of 15 subgraphs is schema-invalid, every PR to every service will fail to compose the supergraph.

Consider the scenario: Football Agent is mid-sprint adding player statistics to the schema. Their branch has a work-in-progress schema that passes typecheck but breaks composition. Every other agent's PR now fails CI. In a 12-week programme with 8 agents working in parallel, this coordination overhead is significant.

The ADR does not address:
- How subgraph schemas are registered in CI without running all 15 services
- Whether SDL files are checked into the repository (the only practical approach for static composition)
- The workflow when one service is intentionally breaking the schema temporarily during development

**Required action:** Define the subgraph SDL registration strategy before Sprint 1. Recommended: each service checks in its `schema.graphql` SDL file. Composition runs against checked-in SDLs, not running services. `rover dev` for local development only.

---

#### FINDING 003-B — ADVISORY: Apollo Router licensing and cost not assessed for production

Apollo Router is Apache-licensed (free). However:

- Apollo GraphOS (managed supergraph, schema registry, usage metrics) starts at $449/month for the Serverless plan. Without GraphOS, there is no managed schema registry — schema composition must be done from static SDL files in CI.
- Apollo Router does not include per-resolver performance metrics without Apollo GraphOS or custom OpenTelemetry instrumentation.
- The ADR does not specify whether Apollo GraphOS is in scope or whether the team will self-manage schema composition from SDL files.

**Required action:** Decide: Apollo GraphOS (~$449-1200/month) or self-managed SDL files + custom observability. Document the decision in ADR-003.

---

#### FINDING 003-C — ADVISORY: GraphQL subscriptions at matchday scale are under-specified

The ADR mentions subscriptions need "ALB sticky sessions or Redis pub/sub" but does not specify which approach is chosen or how it is implemented.

ALB sticky sessions break horizontal scaling: a fan's WebSocket is pinned to one task. If that task is replaced during a deployment (very likely on matchday when fan traffic is highest), the connection drops and the fan loses live match updates. This is visible product damage.

Redis pub/sub is the correct approach but requires the gateway to subscribe to Redis channels and push to connected WebSocket clients. This is an additional implementation concern not currently in any agent's work package.

**Required action:** Add subscription architecture to ADR-003: specify Redis pub/sub as the subscription backend, assign implementation to Platform Agent, include in Sprint 1 scope for match centre feature.

### ADR-004: Kafka / Event Strategy

#### FINDING 004-A — BLOCKING: MSK Serverless cluster cost is significantly understated

The ADR estimates MSK Serverless at "$10/month at dev load." This is incorrect.

AWS MSK Serverless pricing in af-south-1 (prices are approximately 15–20% higher than us-east-1):

- Per cluster-hour: ~$0.87/cluster-hour (af-south-1 rate estimate based on published pricing)
- Per 24 hours: ~$20.88/day
- Per month (3 clusters: dev, staging, prod): ~$1,900/month

Even in the most optimistic scenario (one shared cluster with strict topic isolation per environment), the cost is:

- $0.87 × 24 × 30 = **$626/month minimum for a single always-on MSK Serverless cluster**

This is not "$10/month." The MSK Serverless cost alone will consume 30-40% of the estimated production infrastructure budget.

**Additionally:** MSK Serverless does NOT scale to zero. It runs 24/7 regardless of message volume. The "scales to zero" claim in the ADR is incorrect — MSK Serverless has no idle scaling. Only the per-message cost scales, but the cluster cost is fixed.

**Required action:** Revise cost model. Evaluate: (a) Single MSK cluster shared across environments with topic prefixes (`dev.identity.user.registered`) — reduces cluster cost by 2/3; (b) MSK Serverless only in prod, Kafka in Docker for dev/staging (already in docker-compose.yml); (c) Consider whether Kafka is necessary at Phase 1 scale at all (see FINDING 004-B).

---

#### FINDING 004-B — ADVISORY: Kafka is architectural over-engineering for Phase 1

The case for Kafka is correct for Phase 3+ (2M users, 50K concurrent, real-time scoring). The case for Kafka in Phase 1 (10K users, Year 1) is not made in the ADR.

The Outbox Pattern alone requires each of 15 services to implement:
- An `outbox_events` table
- A polling worker (background job)
- DLQ handling
- Retry logic with exponential backoff
- Dead-letter monitoring
- Idempotent event consumers

That is 15 × (polling worker + DLQ + retry) = 75 individual pieces of Kafka infrastructure. None of it delivers user-facing value. All of it must be built, tested, and operated within the 12-week window.

**The honest alternative for Phase 1:** HTTP webhooks or simple PostgreSQL-backed async queues (e.g., PgBoss) for the 3 critical event flows at launch:

1. `identity.user.registered` → Create loyalty account + wallet + fan profile  
2. `football.match.finished` → Score fantasy, settle GTS  
3. `loyalty.points.awarded` → Update wallet balance

These three flows can be implemented with a simple job queue in < 1 week. The Kafka outbox for all 15 services takes 4–6 weeks.

**This is not a recommendation to remove Kafka.** It is a recommendation to phase it: implement Kafka properly for the 3 critical flows in Sprint 1, stub the rest as synchronous HTTP in Phase 1, migrate to full Kafka in Phase 2 with proper tooling.

**Required action:** Decide whether all 15 services implement the full Outbox Pattern in Phase 1, or whether 3 priority services implement Kafka and the remainder use synchronous HTTP/job-queue internally. Revise sprint plan accordingly.

#### FINDING 004-C — ADVISORY: The Pact.js contract test implementation is not a real contract test

The contract test in the ADR:

```typescript
const parsed = UserRegisteredPayloadSchema.safeParse(event.payload);
expect(parsed.success).toBe(true);
```

This is a schema validation test, not a Pact contract test. A Pact consumer-driven contract test works as follows:

1. **Consumer** (e.g., Loyalty Service) writes a test declaring: "I expect to receive a message with these fields in this shape"
2. **Pact Broker** stores the consumer's contract
3. **Provider** (Identity Service) runs provider verification against the stored contract, proving it publishes messages the consumer can parse

The current implementation has no consumer tests, no Pact Broker, and no provider verification. It is effectively a duplicate of the Zod schema unit test.

**Required action:** Either implement Pact correctly (consumer tests in consumer service, Pact Broker required), or replace with a simpler but honest approach: schema validation tests in `packages/event-schemas` plus integration tests that verify actual Kafka messages against the schema. Remove the "Pact.js" claim from ADR-009 if real Pact is not implemented.

### ADR-005: Database Strategy (Prisma + Aurora Serverless v2)

#### FINDING 005-A — BLOCKING: 15 Aurora Serverless v2 clusters is not viable at Year 1 budget

The ADR mandates "each service has its own Aurora cluster." Fifteen separate Aurora Serverless v2 clusters in production means:

**Cost calculation (production, minimum ACU per service):**
- 15 clusters × 0.5 ACU minimum (when active) × ~$0.12/ACU-hour (af-south-1) × 24 × 30
- = **$648/month minimum for idle production databases alone**
- Dev environment (15 clusters): ~$400/month
- Staging environment (15 clusters): ~$400/month
- **Total minimum database cost: ~$1,450/month across all environments**

At Year 1 revenue of R2M (~$110,000/year), spending $17,400/year (~R315,000) in database infrastructure alone — before a single user registers — is 16% of revenue on databases.

**However:** The ADR conflates "database isolation" (correct) with "cluster isolation" (unnecessary). POPIA and domain boundary requirements are satisfied by:
- Separate PostgreSQL databases (different schemas/databases within one cluster)
- Per-service database users with no cross-database permissions
- Network-level isolation via security groups (database user cannot read another service's tables)

You can have complete data isolation with 2–3 Aurora clusters (e.g., `psl-core-cluster` for 5 high-traffic services, `psl-secondary-cluster` for 10 lower-traffic services) rather than 15.

**Required action:** Revise ADR-005 to specify a cluster topology that meets isolation requirements without 15 independent clusters. Recommended: 2–3 Aurora clusters with per-service databases and users. This reduces database cost by ~75%.

---

#### FINDING 005-B — ADVISORY: Prisma's immutability enforcement is convention, not constraint

The ADR says immutable tables (financial transactions, consent records, wallet ledger) are enforced by "no Prisma update() calls + DB trigger." The DB trigger is not implemented anywhere in the scaffold. The "no Prisma update() calls" is a convention that will be violated the first time an agent or developer makes a mistake.

At a minimum, financial transaction tables and consent records must have database-level constraints enforcing immutability, not application-level convention:

```sql
-- PostgreSQL trigger enforcing immutability
CREATE RULE no_update_wallet_transactions AS
  ON UPDATE TO wallet_transactions DO INSTEAD NOTHING;
```

**Required action:** Add database-level immutability constraints to the Prisma migration for ConsentRecord, LoyaltyTransaction, and WalletTransaction tables. This is a legal/compliance requirement, not an optional enhancement.

The ADR's dev environment estimate of "$80-120/month" does not account for:

|---|---|---|
| ECS Fargate (15 services, 2 tasks each, 0.25 vCPU) | included ($40) | $108/month |

| MSK Serverless (1 cluster) | included ($10) | $626/month |
| ElastiCache Serverless (Redis) | not mentioned | $30-60/month |
| NAT Gateways (2× per AZ, 3 AZs) | not mentioned | $194/month |

This materially changes the programme's financial case. The Year 1 infrastructure cost could represent 15-30% of Year 1 revenue.

#### FINDING 006-B — ADVISORY: Disaster Recovery strategy is a placeholder

The ADR specifies "failover region: eu-west-1 (Ireland)" but provides no design for how DR actually works:

- How is data replicated to eu-west-1? (Aurora Global Clusters? Point-in-time backups? MSK cross-region replication?)
- What is the RTO/RPO for eu-west-1 failover? (The EAB specifies RTO 4 hours, RPO 15 minutes)
- Who triggers a DR failover and how?
- POPIA concern: personal data stored in eu-west-1 (outside South Africa) may require the Information Regulator's approval or explicit disclosure in the privacy policy.

Storing South African citizens' personal data in Ireland for DR purposes may itself require POPIA compliance measures. This intersection of POPIA and DR is not addressed.

**Required action:** Define the DR architecture before go-live. At minimum: Aurora automated backups + S3 cross-region replication of backups to eu-west-1. Aurora Global Clusters add latency and cost. Evaluate whether the 4-hour RTO can be met with backup restore alone (likely yes for Phase 1 scale).

### ADR-007: Football Data Provider

#### FINDING 007-A — ADVISORY: Manual data entry operational model is not specified

The ADR requires "manual override admin endpoints" but doesn't specify who operates them, with what SLA, or what the process is for a live match with goals being scored.

PSL matchdays: typically 2-3 games per weekend, with some midweek rounds. A South African Top 8 knockout match or championship decider will have fans expecting real-time score updates. A 15-minute data delay from API-Football plus a manual entry dependency creates a compounding latency problem.

**The actual user experience risk:** It's 87 minutes, Chiefs are 1-0 up. The match finishes. API-Football publishes the result 15 minutes later. Until then, the match centre shows "LIVE" with a stale score. GTS predictions can't be settled. Fantasy scoring can't start. Fans who check the result immediately after the final whistle see wrong or stale data.

For a launch product, this will generate significant negative feedback and support volume.

**Required action:** Define an operational process: who monitors the match centre during live games? What is the SLA for manual result entry after final whistle? Add this to the operational runbook, not just the ADR.

---

#### FINDING 007-B — ADVISORY: API-Football PSL data quality is unverified

The ADR states API-Football has "partial" PSL coverage. "Partial" is undefined. Before committing to this provider:

- Does API-Football have all 16 PSL clubs?
- Does it have all registered players with consistent player IDs?
- Does it have PSL statistics (clean sheets, saves, yellow/red cards)?
- Does the player ID system match what the Fantasy Agent will use for squad construction?

A fantasy platform where 20% of PSL players have missing statistics is not viable. This must be verified before Sprint 1 begins.

**Required action:** Complete an API-Football data quality audit against PSL data before Sprint 1. Specifically: verify player count per club, statistics field availability, and historical result accuracy for the 2025/26 season. This is a go/no-go gate for the football data provider decision.

### ADR-008: Frontend State Management (TanStack Query + Zustand)

#### FINDING 008-A — ADVISORY: Two GraphQL clients in one application creates cache inconsistency

The ADR uses TanStack Query for GraphQL queries/mutations and Apollo Client for subscriptions only. This means:

- A `football.match.finished` subscription arrives via Apollo Client
- The Apollo Client local cache updates (if the schema is set up for it)
- But the TanStack Query cache for `queryKeys.football.fixtures` is stale and doesn't know the match finished
- The UI shows the old "LIVE" status until TanStack Query's staleTime expires or a manual `invalidateQueries` is called

The ADR does not address how Apollo Client subscription events trigger TanStack Query cache invalidations. Without an explicit bridge, the UI will show inconsistent state during live matches — precisely when consistency matters most.

**Required action:** Define the subscription → cache invalidation bridge. Recommended: Apollo Client subscription callback calls `queryClient.invalidateQueries(queryKeys.football.fixture(fixtureId))` on receipt of relevant events.

---

#### FINDING 008-B — ADVISORY: Server Component QueryClient instantiation is incorrect

The ADR's hydration pattern:

```typescript
export default async function FixturesPage() {
  const queryClient = new QueryClient(); // ← creates new instance per request
  await queryClient.prefetchQuery(...);
```

Creating a new `QueryClient` on every page render is correct for server components (each request needs an isolated cache). However, the ADR does not address that `new QueryClient()` in server components should use the same default options as the client-side `QueryClient`. If they diverge, cached data on the server has different staleTime/retry settings than the client expects.

Minor but worth fixing before it causes subtle SSR/hydration mismatches.

**Required action:** Create a factory function `createServerQueryClient()` that returns a `QueryClient` with the same `defaultOptions` as the client-side instance. Use this factory in all server component prefetch patterns.

### ADR-009: Testing Strategy

#### FINDING 009-A — ADVISORY: Coverage threshold is too low for critical services

80% line coverage is specified uniformly across all services. This is appropriate for peripheral services (Sponsor, Analytics, Search) but is dangerously low for:

- **Wallet Service:** Financial integrity requires 95%+ coverage. A bug in balance calculation or transaction immutability is a platform-integrity incident.
- **Identity Service:** Authentication bypass or POPIA compliance failures require 95%+ coverage.
- **Fantasy Service:** Scoring calculation errors affect competitive fairness and user trust. 90%+.
- **GTS Service:** Settlement errors create direct user-visible unfairness. 90%+.

**Required action:** Revise ADR-009 to specify tiered coverage thresholds. Suggested:

| Service Category | Threshold |
|---|---|
| Wallet, Identity, GTS Settlement | 95% |
| Fantasy scoring engine | 90% |
| All other services | 80% |

#### FINDING 009-B — ADVISORY: K6 load test target does not match platform scale requirement

The load test targets 10K concurrent users. The EAB specifies 50K peak concurrent as the platform requirement. The load test does not validate the platform against its own stated requirement.

**Required action:** Revise K6 target to 50K concurrent. Accept this test may not pass by Week 9 (first test date) and plan an iterative approach: 10K by Week 9, 25K by Week 10, 50K by Week 11.

#### FINDING 009-C — ADVISORY: Pact Broker infrastructure is not planned

The ADR mandates Pact contract tests but does not specify where the Pact Broker runs. PactFlow (managed) costs $0/month for < 5 integrations — insufficient for 15 services. Self-hosted Pact Broker requires an additional service to deploy, monitor, and maintain.

This is not in any agent's work package or the infrastructure Terraform modules.

### ADR-010: Security & POPIA

#### FINDING 010-A — BLOCKING: Minor user processing without guardian consent is a POPIA violation from day 1

The ADR defers guardian consent for under-18 users to "Phase 2." POPIA Section 35 requires that a responsible party must obtain consent from a competent person (parent or guardian) before processing the personal information of a child (under 18).

South African football has a significant under-18 fan demographic. If an under-18 registers, creates a profile, earns points, and makes GTS predictions, PSL One has processed their data without lawful basis from day 1.

Deferring this to Phase 2 is not acceptable. The legal exposure begins at first user registration.

**Required action:** Before go-live, implement one of:
1. Age verification at registration. Users under 18 are blocked from registration pending guardian consent.
2. A "guardian email" flow at registration for users indicating they are under 18 (guardian receives an email, must click an approval link).
3. Legal opinion confirming an alternative lawful basis for processing under-18 data.

This is a legal blocker, not a feature request.

---

#### FINDING 010-B — ADVISORY: `SameSite=Strict` breaks external link entry flows

The ADR specifies `SameSite=Strict` for refresh token cookies. This will break the following user flow:

1. Fan receives a "Your fantasy team scored 84 points!" email
2. Fan clicks the link from Gmail/Outlook
3. Browser navigates to `https://pslone.co.za/fantasy/results`
4. **The `SameSite=Strict` refresh token cookie is NOT sent** (cross-site navigation)
5. Fan is logged out even though they have a valid session

`SameSite=Strict` prevents cookies from being sent on any cross-site navigation, including top-level link clicks from external origins. For a platform that relies on email notifications as a retention mechanism, this is a significant UX regression.

**Required action:** Change refresh token cookie to `SameSite=Lax`. This still blocks CSRF on `POST`/`PUT`/`DELETE` requests (the practical attack surface) while allowing top-level navigation from email links. `SameSite=Lax` is the standard recommendation for session cookies.

---

#### FINDING 010-C — ADVISORY: Audit log `tenantId` field is missing from the implementation

The AuditLogEntry interface in `packages/logger/src/audit.logger.ts` is missing the `tenantId` field that the security strategy requires. The interface has `userId, action, resourceType, resourceId, changes, ipAddress, correlationId` but not `tenantId`.

This is a scaffold bug introduced during implementation. POPIA compliance depends on being able to audit all processing within a specific organisational context.

**Required action:** Fix the AuditLogEntry interface in `packages/logger/src/audit.logger.ts` to include `tenantId`. Apply immediately — this is already in production-bound code.

---

#### FINDING 010-D — ADVISORY: "Active users: indefinite" data retention may violate POPIA purpose limitation

The ADR specifies "Active users: indefinite" data retention. POPIA Section 14 requires data to be deleted or anonymised when it is no longer necessary for the purpose for which it was collected.

For an active fan user, "necessary for the purpose" is ongoing (the fan is actively using the platform). However:

- A fan who registered but never returned after Day 1 is "active" (account exists) but arguably no longer engaging with the purpose. No automatic cleanup is specified for dormant accounts.
- Profile data collected on registration (province, club preference) may no longer be accurate after 3 years. Retaining stale data that was never updated may itself be a POPIA risk.

**Required action:** Define a dormant account policy. Suggested: accounts with no login for 24 months receive a "data retention review" email. Accounts that don't respond are anonymised after 90 days. This is a business decision but must be made and documented before go-live.

### CROSS-02 — ADVISORY: No cross-service distributed transaction strategy

Seven ADRs reference events being published and consumed across services, but no ADR addresses what happens when a distributed workflow fails partway through.

Example: Fan registers → Identity creates user → Loyalty should create account → Wallet should create wallet → Fan should create profile. If the Loyalty service is down when `identity.user.registered` fires, the fan has no loyalty account until the message is replayed. If MSK DLQ is hit (3 failures), the message is dead. The fan has a Cognito account, a ConsentRecord, but no loyalty account, no wallet, no fan profile — permanently.

The Outbox Pattern ensures events are published. It does not ensure events are processed. Dead-lettered messages require manual intervention or a compensating workflow.

**Required action:** Define a policy for DLQ messages. Minimum: (a) an alert fires when any DLQ receives a message; (b) the responsible agent documents a recovery procedure for each consumer; (c) idempotent replay is supported for all consumers.

### CROSS-04 — ADVISORY: No observability strategy defined

No ADR addresses observability. The agent workstream map mentions "OpenTelemetry, AWS X-Ray, Grafana Cloud, CloudWatch" but there is no ADR for the observability strategy.

For a platform with 15 services, 45+ Kafka topics, and matchday spikes, observability is critical. Without it: when a fan reports "my fantasy score is wrong," diagnosing the root cause across Identity → Football → Fantasy → Kafka → Wallet is a multi-day investigation.

**Missing:** A structured logging format standard, distributed trace propagation spec, alert thresholds per service, SLI/SLO definitions, and Grafana dashboard specs.

**Required action:** Add ADR-011: Observability Strategy. This should cover: structured logging format, trace propagation (correlationId passing across Kafka events and HTTP calls), alert thresholds, and which Grafana dashboards must exist before go-live.
