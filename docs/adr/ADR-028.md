# ADR-028: ECS Fargate Runtime and Immutable Container Image Strategy

**Date:** 2026-06-15  
**Status:** Accepted  
**Story:** S3-INFRA-01  

---

## Context

S3-INFRA-01 prepares PSL One for a production-equivalent staging deployment. The accepted baseline audit found that the repository has local Docker Compose infrastructure, IAM policy documents, empty Terraform module directories, and a stale deployment workflow that targets future `services/*` stubs rather than the active applications.

ADR-011 accepted an EC2 + Docker Compose bootstrap approach for an earlier cost-constrained phase. That decision remains useful historical context, but the approved S3-INFRA-01 staging runtime is now ECS Fargate for the current modular monolith API and the Next.js web application.

The active deployable workloads are:

- `apps/api` — NestJS modular monolith
- `apps/web` — Next.js web application

The following are not staging workloads for this story:

- `services/*`
- `apps/admin`
- `apps/club-portal`
- `apps/sponsor-portal`
- Kafka/MSK scaffolding
- identity federation stubs

## Decision

S3-INFRA-01 uses the following staging runtime and deployment model:

- `apps/api` runs on ECS Fargate.
- `apps/web` runs on ECS Fargate.
- Container images are stored in private ECR repositories.
- Every deployed image uses an immutable full Git SHA tag and records the image digest in a release manifest.
- Application traffic enters through an Application Load Balancer.
- PostgreSQL is private through RDS.
- Runtime secrets are referenced from AWS Secrets Manager.
- Logs and basic metrics use CloudWatch.
- GitHub Actions uses OIDC to assume scoped AWS roles.
- ECS services use rolling deployments with deployment circuit breakers and automatic rollback.
- Previous task definition revisions are retained for rollback.
- A separate one-off ECS migration task runs `prisma migrate deploy` before service rollout.
- API readiness is checked before web rollout.
- Post-deployment smoke tests gate completion.
- Initial private-subnet egress uses a configurable NAT gateway count, defaulting to one NAT gateway for cost-conscious staging. VPC endpoints remain a future optimisation option.
- ALB routing is host-based, with configurable API and web hostnames.

This ADR supersedes ADR-011 only for the S3-INFRA-01 staging compute and deployment target. ADR-011 remains accepted historical bootstrap context for the earlier EC2 + Docker Compose approach.

## Scope Boundaries

This story authors code, configuration, Terraform, and documentation only. It does not:

- run AWS authentication
- run `terraform plan`
- run `terraform apply`
- push images to ECR
- deploy ECS services
- create RDS resources
- change DNS
- provision certificates
- activate the PSL season
- implement STORY-40

## Consequences

### Positive

- The staging topology matches the target production container runtime more closely than EC2 bootstrap.
- API and web images become repeatable deployment units.
- Staging deploys can roll back through ECS task definition revisions.
- The migration task is isolated from API startup and can fail independently before traffic shifts.
- GitHub Actions no longer depends on stale `services/*` deployment assumptions.

### Negative

- ECS Fargate, ALB, CloudWatch logs, ECR storage, NAT choices, and RDS can introduce recurring cost. The default NAT gateway count is one for staging cost control and may be increased after availability review.
- The existing Sprint 0 IAM deny guardrails include `DenyECSFargate`, so IAM activation requires explicit review and approval.
- Multi-replica API deployment would make current in-process auth throttling insufficient for distributed traffic.

## Alternatives Considered

| Alternative | Decision |
|-------------|----------|
| Keep ADR-011 EC2 + Docker Compose for staging | Rejected for S3-INFRA-01. The approved staging target is ECS Fargate. |
| Deploy `services/*` stubs | Rejected. The current product runtime is `apps/api` plus `apps/web`. |
| CodeDeploy blue/green | Deferred. ECS rolling deployment with circuit breaker is the approved S3-INFRA-01 strategy. |
| Add Kafka/MSK or Redis/ElastiCache | Rejected for this story. ADR-027 defers durable event infrastructure until measured triggers are met. |
| Make CloudFront mandatory | Rejected for initial staging. CloudFront remains optional and `PLANNED_AFTER_INITIAL_STAGING`. |

## Security Requirements

- No long-lived AWS access keys in GitHub Actions.
- No `.env` or secret values committed.
- ECS task roles are separate from GitHub deploy roles.
- Task execution roles can pull from ECR and write logs only as needed.
- Runtime task roles can read only approved staging secrets.
- RDS is private, encrypted, and inaccessible from public networks.
- Only API and migration task security groups may access PostgreSQL.
- Existing production safety guardrails must not be weakened silently.

## Revisit Triggers

- Staging cost estimate exceeds the approved budget.
- The API must run more than one task replica for authentication-heavy traffic.
- A production deployment story is approved.
- CodeDeploy blue/green becomes required by release policy.
- Kafka/MSK or Redis/ElastiCache revisit triggers from ADR-027 are met.
