# ADR-027 — Durable Event Processing Deferred Pending Measured Requirement

**Status:** Proposed  
**Date:** 2026-06-14  
**Story:** S3-INFRA-00 (Sprint 3 Infrastructure Story 0 — Security & Performance Hardening Gate)  
**Deciders:** Platform engineering, architecture  
**Supersedes:** None  
**Related:** ADR-026 (PSL Season Activation Deferred)

---

## Context

During S3-INFRA-00 triage, finding F-17 identified that the `kafka-client` package exists in the monorepo but is not wired into the API application. All domain side effects (notifications, achievements, activity feed entries, campaign triggers) use direct in-process calls with `.catch(() => null)` guards.

The Technical Review Board classified this as requiring an architecture decision. This ADR records the decision taken for the beta stage and defines the conditions under which the decision must be revisited.

### Current state

- `packages/kafka-client/` exists as infrastructure scaffolding. It is not imported by any API module.
- Domain events are fire-and-forget: `service.doThing().catch(() => null)`.
- If a notification or activity-feed write fails, the failure is silently swallowed. The primary domain operation (prediction, transfer, goal) succeeds regardless.
- This is acceptable for beta where a single process runs, data volumes are small, and fan experience degradation from a missed notification is tolerable.

### Forces

1. **Beta timeline** — Adding a Kafka consumer + producer + schema registry + offset management before staging deployment adds weeks and introduces new infrastructure failure modes.
2. **Measured need** — No measured throughput data exists yet. The assumption that Kafka is required is based on design ceiling (2M fans), not observed load.
3. **Delivery guarantees** — At-least-once delivery for notifications, leaderboard updates, and campaign triggers is desirable but not contractually required for beta.
4. **Settlement durability** — Prediction settlement is currently synchronous and transactional. If settlement itself fails, the transaction rolls back. There is no silent data loss. However, if it succeeds and a subsequent notification fails, the notification is silently dropped.

---

## Decision

**Defer Kafka and all durable event bus infrastructure until a measured trigger is reached.**

For the beta stage:
- Direct in-process calls with `.catch(() => null)` guards remain as-is.
- No Kafka producer or consumer is introduced.
- No Redis pub/sub is introduced as an event system.
- No outbox pattern is implemented.

This is a deliberate, time-bounded decision — not an indefinite deferral.

---

## Consequences

### Positive

- No new infrastructure dependencies for staging.
- Staging deployment can proceed immediately.
- Simpler failure model: a process restart recovers everything from the DB.

### Negative / Accepted risks

| Risk | Mitigation | Revisit trigger |
|------|-----------|----------------|
| Missed notifications when a side-effect call fails | `.catch(() => null)` logs are observable; fans can refresh | Any fan-facing delivery SLA |
| Settlement notification drops | Primary settlement is transactional; notification is best-effort | > 1% drop rate on settlement events |
| Multi-replica divergence | Single ECS replica at beta | Multi-replica ECS deployment |
| Guaranteed campaign trigger delivery | Campaign triggers use same pattern | Real money campaigns enabled |
| Fan Value Ledger write failure silently lost | FVL writes are in-transaction; notification is separate | FVL SLA requirement |

---

## Revisit Triggers

This decision **must be revisited** when any of the following is true:

1. The API runs as more than one ECS task replica simultaneously.
2. Real-money wallet operations are enabled (requires guaranteed callback delivery).
3. Provider webhooks (push notification provider, SES, payment provider) need guaranteed at-least-once delivery.
4. Settlement durability SLA is formalised (e.g., "all settled predictions must trigger a notification within N seconds").
5. Measured event throughput under matchday simulation exceeds what a single process can handle synchronously without exceeding p95 latency targets.

---

## Rejected alternatives

| Alternative | Reason rejected |
|-------------|----------------|
| Wire Kafka immediately | Adds infrastructure complexity before measured need; delays staging |
| Redis pub/sub as lightweight bus | Redis not yet in the stack; same dependency as Kafka without durability benefit |
| Outbox pattern in PostgreSQL | Requires dedicated poller process; equivalent complexity to Kafka consumer for beta |
| Synchronous delivery with retry | Would block request path for notification failures; degrades fan experience more than silent drop |
