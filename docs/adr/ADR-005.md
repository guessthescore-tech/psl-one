# ADR-005: Database Strategy

**Date:** 2026-06-08
**Status:** ACCEPTED
**Owner:** All service agents
**Reviewed By:** Programme Director Agent, Technical Review Board

## Context

PSL One has 15 microservices, each requiring its own isolated data store. The platform uses PostgreSQL-compatible databases. Schema migrations must be version-controlled and repeatable. The ORM must be TypeScript-first, integrate with NestJS, and support Prisma-level developer ergonomics.

CLAUDE.md mandates domain boundaries. No service may access another service's database. This is enforced at the network level (security groups) and must also be enforced by convention.

## Decision

**Prisma ORM + AWS Aurora Serverless v2 (PostgreSQL-compatible) in `af-south-1`.**

- ORM: Prisma (one `schema.prisma` per service, auto-generated typed client)
- Database: AWS Aurora Serverless v2 PostgreSQL in `af-south-1`
- Migrations: Prisma Migrate (version-controlled in each service's `prisma/migrations/`)
- Naming: `snake_case` for tables/columns, `PascalCase` for Prisma model names
- Isolation: each service has its own Aurora cluster and database user

## Options Considered

### ORM

| Option | Decision |
|---|---|
| **Prisma** | CHOSEN — TypeScript-first, auto-generated typed client, excellent migration tooling, Prisma Studio for ops |
| TypeORM | Rejected — verbose, migration management harder, decorator-based typing less safe |
| Drizzle ORM | Deferred — promising but newer, less ecosystem maturity for Phase 1 risk tolerance |
| MikroORM | Rejected — DDD-friendly but steeper learning curve slows agents |

### Database Hosting

| Option | Decision |
|---|---|
| **Aurora Serverless v2** | CHOSEN — auto-scales, af-south-1 available, compatible with PostgreSQL 15, scales to zero |
| RDS PostgreSQL provisioned | Rejected — running cost even at zero load, over-provisioned for Phase 1 |
| Neon (serverless Postgres) | Rejected — not AWS-native, data residency unclear for POPIA |

## Consequences

### Positive
- Prisma generates fully typed query client — no raw SQL for CRUD operations
- `prisma migrate dev` in development, `prisma migrate deploy` in CI/CD
- Aurora Serverless v2 scales from 0.5 ACU → 128 ACU automatically — no capacity planning in Phase 1
- Per-service database isolation enforced at Aurora cluster level
- Prisma Studio available for admin data inspection

### Negative
- Aurora Serverless v2 has ~50ms cold start after idle (acceptable — ECS services keep connections warm via connection pool)
- Each service has its own `schema.prisma` — no cross-service schema view (by design)

### Risks
- Aurora v2 cold start impacting first matchday request: mitigated by PgBouncer connection pooling + ECS services keeping at least one connection warm
- Migration drift between environments: mitigated by `prisma migrate deploy` in CI before every deployment

## Implementation Notes

**Database naming:**
```
psl_<service>_dev
psl_<service>_staging
psl_<service>_prod

Examples:
  psl_identity_prod
  psl_football_prod
  psl_fantasy_prod
  psl_loyalty_prod
  psl_wallet_prod
```

**Per-service directory structure:**
```
services/<name>/
  prisma/
    schema.prisma
    migrations/
      YYYYMMDDHHMMSS_initial/
        migration.sql
  src/
    database/
      database.module.ts
      database.service.ts   (wraps PrismaClient)
```

**Standard Prisma schema header:**
```prisma
generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}
```

**Audit fields on all tables:**
```prisma
model Example {
  id        String   @id @default(uuid())
  createdAt DateTime @default(now()) @map("created_at")
  updatedAt DateTime @updatedAt @map("updated_at")
  
  @@map("examples")
}
```

**Connection string format:**
```
DATABASE_URL=postgresql://<user>:<password>@<aurora-endpoint>:5432/psl_<service>_<env>?schema=public&connection_limit=10
```

**Connection pooling:**
- Development: direct connection
- Production: PgBouncer sidecar on ECS task (transaction pooling mode)
- Max connections: 10 per service instance; Aurora Serverless v2 max: 1000

**Financial / audit tables (Identity, Wallet, Loyalty) must use:**
```prisma
// Soft delete pattern for POPIA right-to-erasure compliance
deletedAt DateTime? @map("deleted_at")
// Immutable after creation — no UPDATE allowed on these rows
// Enforced by: no Prisma update() calls + DB trigger
```

**Cross-service data access rule:**
Any service that needs data owned by another service MUST:
1. Subscribe to the relevant Kafka events and maintain a read-only local copy, OR
2. Call the owning service's GraphQL subgraph or REST API

Never access another service's database directly. This is enforced by security group rules in Terraform.
