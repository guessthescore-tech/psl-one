# ADR-025: BetaLaunchModule Delegates Season Checks to SeasonSwitchingService

**Date:** 2026-06-14  
**Status:** ACCEPTED  
**Story:** STORY-39  

---

## Context

`BetaLaunchModule` needs to display readiness status and run dry-runs based on the 13-check season activation gate. `SeasonSwitchingModule` already owns these checks. The question was whether `BetaLaunchModule` should duplicate the check logic or delegate.

## Decision

`BetaLaunchModule` imports `SeasonSwitchingModule` and calls `SeasonSwitchingService` directly for readiness checks and dry-run analysis. No check logic is duplicated.

## Alternatives Considered

- **BetaLaunchModule owns a copy of readiness checks** — rejected; creates two sources of truth that will diverge; any new check must then be added in two places
- **BetaLaunchModule calls SeasonSwitchingController via HTTP** — rejected; internal module calls over HTTP adds latency and a circular dependency risk
- **Merge BetaLaunchModule into SeasonSwitchingModule** — rejected; the domains are different (beta cohorts and approval are not season switching concerns); keep bounded context separation

## Consequences

- `SeasonSwitchingModule` is the single source of truth for readiness checks
- Adding a 14th readiness check requires updating only `SeasonSwitchingService`
- `BetaLaunchModule` is lightweight — it adds cohort management, approval records, smoke tests, and runbook on top of the checks
- `SeasonSwitchingModule` must be in `BetaLaunchModule`'s imports array
- Tests for `BetaLaunchService` mock `SeasonSwitchingService` rather than reimplementing checks

## Revisit Triggers

- If `SeasonSwitchingService` readiness checks become async or slow (then consider caching the result)
- If `BetaLaunchModule` needs checks that are *not* in the 13-gate (add a `BetaReadinessService` alongside `SeasonSwitchingService`)
