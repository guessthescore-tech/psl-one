# ADR-012: Points System Separation

**Date:** 2026-06-14  
**Status:** ACCEPTED  
**Story:** Sprint 2 (STORY-19 through STORY-38)  

---

## Context

PSL One has multiple scoring and value systems. Early development risked conflating them — using Fan Value for gameplay, or treating prediction points as redeemable currency.

## Decision

Four strictly separate non-financial systems:

1. **Fan Value** (`FanValueLedger`) — non-financial loyalty engagement score
2. **Prediction Points** (`PredictionPointsLedger`) — Guess the Score leaderboard points
3. **Gameplay Points** (`SocialPredictionPointsEntry`) — Social challenge stakes and awards
4. **Fantasy Points** (`FantasyGameweekScore`) — Fantasy Football scoring

No system can convert into another. No system has monetary value.

## Alternatives Considered

- **Single points table with type column** — rejected because it creates schema coupling between unrelated domains and risks accidental aggregation
- **Fan Value as currency with conversion** — rejected; Fan Value has no monetary value and treating it as redeemable currency would require gambling compliance
- **Fantasy points on FanValueLedger** — rejected; Fantasy points are GW-scoped and sum to a season total that is independent of the engagement score

## Consequences

- Four distinct models with distinct tables
- Each ledger is immutable (append-only)
- Frontend must display them as separate values — no aggregation across types
- Any proposal to merge or convert systems requires a new ADR
