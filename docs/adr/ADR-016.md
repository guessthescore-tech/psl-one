# ADR-016: Direct Service Calls for Side Effects

**Date:** 2026-06-14  
**Status:** ACCEPTED (until Kafka is wired)  
**Story:** STORY-19 (Fan Value), STORY-22 (Notifications), STORY-23 (Activity Feed)  

---

## Context

PSL One's architecture calls for Kafka event publishing. However, Kafka introduces operational complexity. During Sprint 1 and Sprint 2 beta development, spinning up a reliable Kafka consumer infrastructure was not prioritised.

## Decision

Side effects (Fan Value, Notifications, Activity Feed, Campaign Triggers) are implemented as direct synchronous service calls within the originating service. No Kafka producers or consumers are active.

This is explicitly a temporary decision pending load justification.

## Migration Plan

When Kafka is approved (DECISION_REQUIRED in Sprint 3):
1. Each domain service publishes a domain event to a named Kafka topic
2. Side-effect handlers consume those topics
3. Direct calls are removed

## Alternatives Considered

- **Kafka from day one** — rejected; operational complexity of reliable Kafka infrastructure was not justified for beta scale with a small user cohort
- **Database outbox pattern** — considered; would decouple the write from the side effect without Kafka; deferred because the added complexity is not needed at beta scale
- **Bull/Redis queue** — considered; Redis is already in the stack but queue infrastructure adds worker processes not yet needed

## Consequences

- Side effects are synchronous — a Fan Value write failure will fail the parent request
- No retry or dead-letter queue for side effects
- Tight coupling between originating service and side-effect services
- Acceptable for beta; not acceptable at 2M fan production scale under high load
- Kafka is in `docker-compose.yml` ready for wiring
