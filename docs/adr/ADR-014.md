# ADR-014: Player externalId Non-Uniqueness

**Date:** 2026-06-14  
**Status:** ACCEPTED  
**Story:** STORY-29  

---

## Context

PSL player data comes from external sources. External data providers may reuse player IDs across imports, seasons, or data source changes. An early schema enforced uniqueness on `Player.externalId`, causing import failures when the same player was imported for multiple seasons.

## Decision

`Player.externalId` is NOT a unique constraint. Multiple `Player` records may share the same `externalId`. All queries for a player by external ID must use `findFirst` with a season scope, not `findUnique`.

## Alternatives Considered

- **Keep uniqueness + version the external ID** — rejected; external providers don't version IDs and we can't control the provider's numbering scheme
- **Separate `SeasonPlayer` table instead of `Player`** — rejected; would require migrating all foreign keys that reference `Player.id`
- **Ignore duplicates, last-write-wins** — rejected; causes silent data loss during re-import

## Consequences

- `prisma.player.findFirst({ where: { externalId, seasonTeamPlayers: { some: { seasonId } } } })` is the correct pattern
- `prisma.player.findUnique({ where: { externalId } })` must never be used
- Import pipelines use upsert with composite keys (externalId + seasonId) where possible
- Tests that create players must not assume externalId uniqueness
