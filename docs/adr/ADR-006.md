# ADR-006: AWS Deployment Strategy

**Date:** 2026-06-08
**Status:** ACCEPTED
**Owner:** DevOps Agent
**Reviewed By:** Programme Director Agent, Technical Review Board

## Context

PSL One is AWS-native. The platform requires separate environments (dev, staging, prod), South African data residency for POPIA, zero-downtime deployments, and a container orchestration strategy that minimises operational overhead in the 12-week delivery window.

The platform must handle matchday spikes (up to 50K concurrent users) and return to normal load between matches. Cost control matters: Year 1 revenue target is R2M.

## Decision

**3 AWS accounts (psl-dev, psl-staging, psl-prod) via AWS Organisations + ECS Fargate in af-south-1 + blue/green deployments.**

Primary region: `af-south-1` (Cape Town) — POPIA data residency
Failover region: `eu-west-1` (Ireland) — DR only, no active traffic in Phase 1
Container orchestration: AWS ECS Fargate (serverless containers)
Container registry: AWS ECR (one repository per service)
Secrets: AWS Secrets Manager with automatic rotation

## Options Considered

### Account Structure

| Option | Decision |
|---|---|
| **3 AWS accounts via AWS Organisations** | CHOSEN — security blast radius isolation, billing clarity, independent IAM policies |
| Single account (all environments) | Rejected — security risk, no billing isolation, prod credentials too accessible |
| AWS Organisations + full OU structure | Deferred — enterprise-grade but over-engineered for Phase 1 |

### Container Orchestration

| Option | Decision |
|---|---|
| **AWS ECS Fargate** | CHOSEN — no server management, native AWS integration (ECR, Secrets Manager, CloudWatch, X-Ray), appropriate for Phase 1 scale |
| AWS EKS (Kubernetes) | Rejected — 2–3× operational complexity, $150+/month for control plane even at zero load |
| AWS Lambda | Rejected — cold starts incompatible with API services requiring <300ms p95 latency |

## Consequences

### Positive
- Account separation means a dev mistake cannot affect production
- ECS Fargate scales containers automatically via ECS Service Auto Scaling
- Blue/green deployment via ECS CodeDeploy integration = zero-downtime deployments
- af-south-1 satisfies POPIA data residency — no cross-border data transfer for personal data

### Negative
- 3 AWS accounts increases initial setup time (acceptable — DevOps agent handles this in Sprint 0)
- af-south-1 has fewer available services than us-east-1 (all required services: MSK Serverless, Aurora Serverless v2, ECS Fargate, ElastiCache Serverless are available)

### Risks
- af-south-1 capacity constraints on matchday: mitigated by ECS capacity providers + auto-scaling
- Cross-account ECR access: mitigated by ECR replication or cross-account pull permissions in Terraform

## Implementation Notes

**Account structure:**
```
AWS Organisation root
├── psl-dev   (account ID: to be provisioned)
├── psl-staging (account ID: to be provisioned)
└── psl-prod  (account ID: to be provisioned)
```

**Terraform directory structure:**
```
infra/terraform/
  modules/
    ecs-service/      # reusable ECS task + service module
    aurora-cluster/   # Aurora Serverless v2 cluster
    msk-cluster/      # MSK Serverless cluster
    elasticache/      # Redis cluster
    alb/              # Application Load Balancer
    waf/              # WAF rules
    iam/              # IAM roles and policies
    vpc/              # VPC, subnets, security groups
  environments/
    dev/
      main.tf
      variables.tf
      terraform.tfvars
    staging/
      main.tf
      variables.tf
      terraform.tfvars
    prod/
      main.tf
      variables.tf
      terraform.tfvars
```

**ECS task definition pattern (per service):**
```hcl
resource "aws_ecs_task_definition" "service" {
  family                   = "psl-${var.service_name}-${var.environment}"
  requires_compatibilities = ["FARGATE"]
  network_mode             = "awsvpc"
  cpu                      = var.cpu     # 256 dev, 512 prod
  memory                   = var.memory  # 512 dev, 1024 prod
  
  container_definitions = jsonencode([{
    name  = var.service_name
    image = "${aws_ecr_repository.service.repository_url}:${var.image_tag}"
    
    secrets = [
      { name = "DATABASE_URL", valueFrom = aws_secretsmanager_secret.db_url.arn }
    ]
    
    healthCheck = {
      command = ["CMD-SHELL", "curl -f http://localhost:3000/health || exit 1"]
    }
  }])
}
```

**Blue/green deployment:**
```yaml
# GitHub Actions deployment step
- name: Deploy to ECS
  run: |
    aws ecs update-service \
      --cluster psl-${{ env.ENVIRONMENT }} \
      --service ${{ env.SERVICE_NAME }} \
      --force-new-deployment \
      --deployment-configuration "minimumHealthyPercent=100,maximumPercent=200"
```

**Auto-scaling (production):**
- Scale out: CPU > 70% for 2 minutes → add 2 tasks
- Scale in: CPU < 30% for 5 minutes → remove tasks
- Min tasks: 2 (HA), Max tasks: 20 per service

**Networking:**
- VPC: 10.0.0.0/16 per account
- Public subnets: ALB + NAT Gateways
- Private subnets: ECS tasks, Aurora, MSK, ElastiCache
- No direct public IP on any ECS task

**IAM task execution role (per service):**
- Read own Secrets Manager secrets
- Write to own CloudWatch log group
- Pull from own ECR repository
- Publish to own MSK topics
- No cross-service IAM permissions

**Cost estimates (dev environment, 24/7):**
- ECS Fargate (5 services × 0.25 vCPU / 0.5GB): ~$40/month
- Aurora Serverless v2 (3 clusters, idle): ~$30/month
- MSK Serverless: pay-per-use, ~$10/month at dev load
- Total dev: ~$80–120/month
