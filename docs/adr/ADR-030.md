# ADR-030: Structured API Logging, Request Correlation, and Cache Coherence

**Date:** 2026-06-29  
**Status:** Accepted  
**Story:** S3-INFRA-00  

## Context

PSL One had the logging shape described in `docs/operations/OBSERVABILITY-REQUIREMENTS.md`, but the API was still emitting sentence-style logs and `console.log` startup output. There was no request-scoped correlation context, so logs from a single fan request could not be tied together across middleware, guards, services, and external adapters. In parallel, the response cache interceptor had no stampede protection, and the existing invalidation service was not wired into write paths.

The auth dev sinks also printed raw reset tokens and verification URLs. That was acceptable neither for shared logs nor for a secure handover.

## Decision

Implement an API-local observability layer with three parts:

1. `RequestContextService` backed by `AsyncLocalStorage` to carry `requestId`, `correlationId`, and `userId` through the request lifecycle.
2. `StructuredLoggerService` as the Nest logger implementation so logs are emitted as JSON objects with timestamp, level, request context, and action fields.
3. `RequestLoggingInterceptor` to record request completion and failure events with method, route, status code, and duration.

Also:

- Add in-flight promise deduplication to the HTTP cache interceptor to prevent cache stampedes on hot keys.
- Wire cache invalidation into the profile, media, campaign, and campaign participation write paths.
- Remove raw token and URL logging from auth development sinks.

## Consequences

### Positive

- Request logs are now searchable and correlatable by `requestId` and `correlationId`.
- `userId` becomes available to request-scoped logs after authentication.
- Log level discipline is enforced at the logger boundary rather than by convention.
- Cache misses for hot read endpoints no longer fan out into duplicate concurrent fetches.
- Fan-visible mutations now invalidate the corresponding cached read surfaces immediately.
- Auth dev sinks no longer leak raw reset tokens or verification links.

### Negative

- The implementation is API-local and still writes to stdout/stderr rather than to a managed log backend.
- Cache protection is still in-memory; it does not yet solve cross-process cache coherence.
- If the API is scaled horizontally, the cache layer will need a shared backend before the interceptor can be considered distributed-safe.

## Alternatives Considered

| Alternative | Decision |
|-------------|----------|
| Keep sentence-style logs and rely on manual grep | Rejected. That does not meet the observability requirement. |
| Adopt OpenTelemetry and CloudWatch/X-Ray immediately | Deferred. That is a wider infrastructure story and not required to close the current gap. |
| Move response caching directly to Redis now | Deferred. The current task is to make the existing cache coherent and stampede-safe first. |
| Leave auth dev sinks verbose for local debugging | Rejected. The secret leakage risk is not acceptable. |

## Revisit Triggers

- The API moves to multiple replicas and needs shared request-log ingestion and distributed cache coherence.
- OpenTelemetry/X-Ray becomes a release requirement.
- Redis or another shared cache backend is introduced.
- A managed log platform replaces stdout/stderr JSON lines.
