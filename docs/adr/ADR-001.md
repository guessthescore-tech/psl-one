# ADR-001: Authentication Strategy

**Date:** 2026-06-08
**Status:** ACCEPTED
**Owner:** Platform Agent (Identity)
**Reviewed By:** Programme Director Agent, Technical Review Board

## Context

PSL One requires a robust, scalable authentication system for fan registration, login, MFA, social auth and admin access. The platform targets 2 million registered fans with South African data residency requirements under POPIA. The auth system is the first service built and every other service depends on it.

Requirements:
- Fan registration via email + mobile (OTP)
- Social login (Google, Apple)
- JWT-based stateless auth across all microservices
- RBAC roles: FAN, CLUB_ADMIN, SPONSOR_ADMIN, PSL_ADMIN, SUPER_ADMIN, COMPLIANCE_OFFICER
- POPIA-compliant consent capture at registration
- Scale to 2M users without architectural change
- South African data residency (af-south-1)

## Decision

**Use AWS Cognito as the authentication provider.**

Supplemented by a `consent_records` table in the Identity service database to handle POPIA granular consent purposes that Cognito does not natively track.

## Options Considered

### Option A: AWS Cognito
Pros: Native AWS integration, af-south-1 data residency, free to 50K MAU, Lambda triggers for customisation, social federation (Google/Apple), scales to 2M without change.
Cons: Limited custom login UI, migration complexity if switching later, POPIA consent requires supplementary table.

### Option B: Auth0
Pros: Best-in-class DX, excellent social login, rich customisation, strong enterprise features.
Cons: ~$0.70/MAU beyond 7,000 free (unaffordable at 400K MAU target: ~R5M/yr), data residency requires explicit configuration, external dependency.

### Option C: Keycloak (self-hosted)
Pros: Full control, no per-MAU cost, POPIA-friendly.
Cons: Significant operational burden, infrastructure management, upgrade complexity — not appropriate for a 12-week build.

### Option D: Supabase Auth
Pros: Simple, PostgreSQL-native, open source.
Cons: Less enterprise-grade, limited MFA options, not AWS-native, data residency unclear.

## Consequences

### Positive
- Zero auth infrastructure to manage in Phase 1
- POPIA data residency satisfied by af-south-1 Cognito user pools
- Free tier covers all of Year 1 (50K MAU free)
- Lambda triggers enable custom post-registration flows (consent capture, fan profile creation event)
- JWT tokens from Cognito verifiable by all services using the public JWKS endpoint

### Negative
- Custom login UI requires Cognito Hosted UI customisation or own UI + Cognito API calls
- POPIA consent purposes must be stored separately (supplementary `consent_records` table)
- If Cognito is ever replaced, user pool migration is complex

### Risks
- Cognito service availability: mitigated by AWS SLA (99.9%)
- POPIA gap in Cognito: mitigated by supplementary consent model (see Implementation Notes)

## Implementation Notes

**Cognito User Pool configuration:**
- Region: `af-south-1`
- Password policy: min 8 chars, uppercase, lowercase, number, special char
- MFA: optional (TOTP or SMS), required for CLUB_ADMIN and above
- User attributes: `email` (required, verified), `phone_number` (required, verified via OTP), `custom:province`, `custom:primaryClub`, `custom:pslRole`

**RBAC implementation:**
- Cognito Groups map to RBAC roles: `fans`, `club-admins`, `sponsor-admins`, `psl-admins`, `super-admins`, `compliance-officers`
- JWT claims include `cognito:groups` — all services validate this claim
- Identity service exposes a `/auth/token/validate` endpoint that all services can use

**POPIA consent flow:**
```
Fan registers → Cognito creates user → Lambda Post-Confirmation trigger fires
→ Lambda calls Identity Service → Identity Service writes ConsentRecord rows:
  { userId, purpose: 'MARKETING', granted: bool, timestamp, ipAddress, version: '1.0' }
  { userId, purpose: 'ANALYTICS', granted: bool, timestamp, ipAddress, version: '1.0' }
  { userId, purpose: 'THIRD_PARTY_SHARING', granted: bool, timestamp, ipAddress, version: '1.0' }
→ Identity Service publishes `identity.user.registered` Kafka event
```

**Token configuration:**
- Access token: 15 minutes
- Refresh token: 30 days
- Token type: JWT RS256 (Cognito signs with RSA, public key available at JWKS endpoint)

**Social login:**
- Google: Cognito Identity Provider federation
- Apple: Cognito Identity Provider federation
- Both map to Cognito user pool automatically

**Required packages:**
```
services/identity:
  amazon-cognito-identity-js
  @aws-sdk/client-cognito-identity-provider
  
packages/auth-guards:
  aws-jwt-verify (for verifying Cognito JWTs without calling Cognito on each request)
```
