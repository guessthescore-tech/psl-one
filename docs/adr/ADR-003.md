# ADR-003: API Strategy

**Date:** 2026-06-08
**Status:** ACCEPTED
**Owner:** Platform Agent (Gateway)
**Reviewed By:** Programme Director Agent, Technical Review Board

## Context

PSL One has a fan-facing web app, admin portals, and potential future mobile apps and partner integrations. The platform has 15 backend microservices, each owning its own data. Fans need to query data that spans multiple services (fixture + standings + fantasy team in one screen). An API strategy must be decided before any service builds its external interface.

Requirements:
- Single entry point for client applications
- Support complex multi-domain queries (fan dashboard queries identity + football + fantasy + loyalty)
- Real-time updates for live match events
- Admin endpoints for tooling and health checks
- Future mobile app compatibility
- No N+1 query storms at the gateway

## Decision

**GraphQL Federation (external) + REST (internal admin + health).**

- External fan/partner API: GraphQL Federation via Apollo Router
- Service-to-service communication: Kafka events (async) or direct HTTP where sync is required
- Admin and health endpoints: REST/OpenAPI
- Real-time (live scores): GraphQL Subscriptions via WebSocket on the gateway
- Webhooks (ticketing/payment callbacks): REST with HMAC verification

## Options Considered

### Option A: GraphQL Federation only
Pros: Unified client API, flexible querying, type system enforces contracts.
Cons: Admin tooling awkward in GraphQL, health check conventions don't fit GraphQL.

### Option B: REST only
Pros: Simple, cacheable, universally understood.
Cons: Multiple round trips for complex queries (fan dashboard = 5+ REST calls), over-fetching, under-fetching problems at scale.

### Option C: GraphQL (external) + REST (internal admin + health)
Pros: Best of both worlds — fan API is flexible and efficient; admin/ops endpoints stay simple and easily curl-able.
Cons: Two API paradigms to maintain (acceptable — they serve different consumers).

### Option D: tRPC
Pros: Type-safe end-to-end, excellent TypeScript DX.
Cons: TypeScript clients only, not suitable for external partners or non-TS consumers, not appropriate for public API.

## Consequences

### Positive
- Apollo Router handles query planning, authorization propagation, and rate limiting at the edge
- Each service owns its GraphQL subgraph — no shared schema file, no coordination bottleneck
- Fan app can fetch exactly what it needs in one request
- Admin REST endpoints remain simple and CI/CD-scriptable

### Negative
- Apollo Router adds an infrastructure component to manage
- Federation supergraph composition must be kept valid — schema changes in one service can break composition
- Subscriptions require sticky connections (ALB sticky sessions or Redis pub/sub)

### Risks
- Supergraph composition failure: mitigated by `rover supergraph compose` in CI (PR blocks if composition fails)
- N+1 at gateway: mitigated by DataLoader pattern in each subgraph resolver

## Implementation Notes

**Service subgraph setup:**
```typescript
// Each service's GraphQL module
import { ApolloFederationDriver } from '@nestjs/apollo';
import { buildSubgraphSchema } from '@apollo/subgraph';

// Every entity type must declare @key
type Fan @key(fields: "id") {
  id: ID!
  username: String!
}
```

**Apollo Router configuration:**
```yaml
# router.yaml
supergraph:
  listen: 0.0.0.0:4000
authentication:
  jwt:
    jwks:
      - url: https://cognito-idp.af-south-1.amazonaws.com/<pool-id>/.well-known/jwks.json
headers:
  all:
    request:
      - propagate:
          named: authorization
```

**Gateway service (`services/gateway`) responsibilities:**
- Apollo Router process (Rust binary, not NestJS)
- Rate limiting (via Apollo Router plugin or AWS WAF upstream)
- Request logging and tracing
- Schema composition validation in CI

**REST admin endpoints** are mounted directly on each NestJS service under `/admin/*` and are not federated through the GraphQL gateway. They are protected by an internal network security group rule (not publicly accessible).

**WebSocket subscriptions:**
- Apollo Router forwards subscription requests to a dedicated subscription handler
- Live match events flow: `football.match.event` Kafka topic → Notification Service → GraphQL subscription push

**Health check convention (all services):**
```
GET /health → { status: 'ok', service: 'identity', version: '1.0.0', timestamp: '...' }
GET /health/ready → { status: 'ok' } (only when DB connection is ready)
```
