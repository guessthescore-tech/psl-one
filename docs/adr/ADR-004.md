# ADR-004: Kafka / Event Strategy

**Date:** 2026-06-08
**Status:** ACCEPTED
**Owner:** Platform Agent
**Reviewed By:** Programme Director Agent, Technical Review Board

## Context

PSL One is event-driven. Fan registration must trigger loyalty account creation. Match completion must trigger fantasy scoring and GTS settlement. Wallet transactions must be auditable and immutable. All of this requires a reliable, durable message bus. CLAUDE.md mandates: "Always publish Kafka events."

Requirements:
- Durable, ordered event delivery
- Schema governance across 15 services
- Reliable publishing (no lost events on service crash)
- TypeScript-native schema validation
- Scalable to 2M fans with matchday spikes
- Minimal operational overhead in Phase 1

## Decision

**AWS MSK Serverless + JSON Schema validated with Zod + topic naming convention `<domain>.<entity>.<action>` + Transactional Outbox Pattern.**

Schema registry: AWS Glue Schema Registry (native MSK integration)
Event schemas stored in: `packages/event-schemas`
Phase 1: MSK Serverless → Phase 3: MSK Provisioned if volume justifies

## Options Considered

### Kafka Deployment

| Option | Decision |
|---|---|
| **AWS MSK Serverless** | CHOSEN — no broker management, scales to zero overnight, pay-per-use appropriate for Phase 1 |
| AWS MSK Provisioned | Deferred to Phase 3 when predictable load makes cost comparison clear |
| Confluent Cloud | Rejected — cost, external dependency |
| Self-hosted Kafka on ECS | Rejected — operational overhead incompatible with 12-week timeline |

### Schema Format

| Option | Decision |
|---|---|
| **JSON Schema + Zod** | CHOSEN — TypeScript-native, readable, validated at compile time and runtime |
| Avro + Confluent Schema Registry | Rejected — Java-origin tooling, less TypeScript-native DX |
| Protobuf | Rejected — complex TS tooling for no material benefit at Phase 1 scale |

## Consequences

### Positive
- MSK Serverless eliminates broker sizing decisions in Phase 1
- Zod schemas in `packages/event-schemas` provide compile-time and runtime type safety
- Transactional Outbox Pattern guarantees no events lost on service crash
- Topic naming convention makes event discovery trivial

### Negative
- MSK Serverless has higher per-message cost than provisioned at high volume (acceptable for Phase 1)
- JSON is larger than Avro/Protobuf (acceptable — Kafka handles this easily)
- Outbox Pattern requires each service to implement a polling worker

### Risks
- Schema evolution breaking consumers: mitigated by additive-only changes + version field in envelope
- MSK cold start latency: mitigated by keeping consumers warm (ECS services always running)

## Implementation Notes

**Topic naming convention:**
```
<domain>.<entity>.<action>

Examples:
  identity.user.registered
  identity.user.deleted
  football.match.started
  football.match.finished
  football.goal.scored
  fantasy.squad.created
  fantasy.gameweek.scored
  gts.prediction.created
  gts.prediction.settled
  loyalty.points.awarded
  loyalty.tier.changed
  wallet.transaction.credited
  wallet.transaction.debited
  notifications.push.sent
  notifications.email.sent
```

**Event envelope (all events must include):**
```typescript
// packages/event-schemas/src/envelope.ts
import { z } from 'zod';

export const KafkaEventEnvelopeSchema = z.object({
  eventId: z.string().uuid(),
  eventType: z.string(),
  version: z.string().regex(/^\d+\.\d+\.\d+$/),
  timestamp: z.string().datetime(),
  tenantId: z.string(),
  correlationId: z.string().uuid(),
  payload: z.unknown(),
});

export type KafkaEventEnvelope<T = unknown> = {
  eventId: string;
  eventType: string;
  version: string;
  timestamp: string;
  tenantId: string;
  correlationId: string;
  payload: T;
};
```

**Example event schema:**
```typescript
// packages/event-schemas/src/identity/user-registered.event.ts
import { z } from 'zod';

export const UserRegisteredPayloadSchema = z.object({
  userId: z.string().uuid(),
  email: z.string().email(),
  mobile: z.string(),
  province: z.string().optional(),
  primaryClubId: z.string().uuid().optional(),
  consentMarketing: z.boolean(),
  consentAnalytics: z.boolean(),
  consentThirdParty: z.boolean(),
});

export type UserRegisteredPayload = z.infer<typeof UserRegisteredPayloadSchema>;
export const USER_REGISTERED_TOPIC = 'identity.user.registered';
export const USER_REGISTERED_VERSION = '1.0.0';
```

**Transactional Outbox Pattern (required for all services):**
```sql
-- Each service database has an outbox table
CREATE TABLE outbox_events (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  topic VARCHAR(255) NOT NULL,
  payload JSONB NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  published_at TIMESTAMPTZ,
  attempts INT NOT NULL DEFAULT 0
);
```
```typescript
// In the same database transaction as the domain write:
await prisma.$transaction([
  prisma.user.create({ data: userData }),
  prisma.outboxEvent.create({
    data: { topic: USER_REGISTERED_TOPIC, payload: event }
  }),
]);
// Background worker polls outbox_events WHERE published_at IS NULL and publishes to MSK
```

**`packages/kafka-client` responsibilities:**
- NestJS Kafka module wrapper with MSK IAM authentication
- Outbox worker base class
- Consumer group management
- Dead letter queue (DLQ) routing after 3 failed attempts

**Consumer group naming:**
```
<service>.<domain>.<entity>
Examples:
  loyalty-service.identity.user
  fan-service.identity.user
  notifications-service.football.match
```

**Partition strategy:**
- Default: 12 partitions per topic
- Key: `tenantId:entityId` for ordering within tenant/entity scope
