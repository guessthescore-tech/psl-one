# ADR-015: Immutable Ledger Pattern

**Date:** 2026-06-14  
**Status:** ACCEPTED  
**Story:** STORY-11 (predictions), STORY-38 (social prediction)  

---

## Context

Points ledgers (`PredictionPointsLedger`, `SocialPredictionPointsEntry`, `AdminAuditLog`) must be auditable. If rows can be updated or deleted, audit trails can be falsified or points balances manipulated.

## Decision

Ledger tables are append-only:
- `PredictionPointsLedger` — never updated or deleted
- `SocialPredictionPointsEntry` — never updated or deleted  
- `AdminAuditLog` — never updated or deleted

Corrections are new entries (e.g., a `VOID_RESTORED` entry on a cancelled challenge rather than deleting the `POINTS_COMMITTED` entry).

## Alternatives Considered

- **Mutable balance field** — rejected; allows silent manipulation and cannot reconstruct history
- **Event sourcing with replay** — considered but over-engineered for beta scale; ledger tables achieve the same auditability without event store infrastructure
- **Soft delete (deleted_at column)** — rejected; a row that is soft-deleted is still retrievable, so it doesn't protect against accidental modification — full append-only enforcement is stricter

## Consequences

- A fan's effective balance is the SUM of all entries, not a mutable balance field
- Historical balances can be reconstructed at any point in time
- Schema migrations must never add `ON DELETE CASCADE` to ledger rows
- Service code must not call `.update()` or `.delete()` on ledger tables
- Leaderboard queries aggregate from ledger entries — no denormalised totals (except for performance caching)
