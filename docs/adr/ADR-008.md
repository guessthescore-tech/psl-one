# ADR-008: Frontend State Management Strategy

**Date:** 2026-06-08
**Status:** ACCEPTED
**Owner:** Frontend Agent
**Reviewed By:** Programme Director Agent, Technical Review Board

## Context

PSL One's fan web app (Next.js 15 App Router) manages two distinct categories of state:

1. **Server state:** Data fetched from the GraphQL API (fixtures, standings, fantasy team, loyalty balance, GTS predictions). This data needs caching, background refresh, optimistic updates, and cache invalidation.

2. **UI state:** Ephemeral local state (modal open/close, form progress, wizard steps, notification banners). This data lives only in the browser and is never server-persisted.

Next.js 15 App Router server components handle initial page renders without client-side fetching. The state management solution must coexist with server components.

## Decision

**TanStack Query v5 (server state) + Zustand (UI state).**

- **TanStack Query v5:** All API/GraphQL data fetching, caching, background refresh, optimistic updates, and pagination
- **Zustand:** Ephemeral UI state (modal state, wizard steps, toast queue, user preferences not persisted to server)
- **Next.js App Router server components:** Used for initial data fetch where possible (no client-side fetch for static/slow-changing data)
- **Apollo Client:** Used alongside TanStack Query *only* for GraphQL subscriptions (live match events) — not for standard queries

CLAUDE.md mandates: "Never store business logic in frontend." All business rules live in backend services.

## Options Considered

### Option A: TanStack Query v5 + Zustand
Pros: Best separation of concerns, excellent DX, lightweight, battle-tested at scale, works well with Next.js App Router.
Cons: Two libraries (acceptable — they serve distinct purposes that don't overlap).

### Option B: TanStack Query only
Pros: One library.
Cons: TanStack Query's `useQuery` state is scoped to components — no clean global UI state management for modals, toasts, and multi-step flows.

### Option C: Redux Toolkit + RTK Query
Pros: Mature, well-understood, strong TypeScript support.
Cons: Verbose, over-engineered for this scale, boilerplate-heavy for AI agents to generate consistently.

### Option D: SWR
Pros: Simple, Next.js-compatible.
Cons: Less feature-rich (no optimistic updates, less control over cache invalidation), Vercel-centric positioning.

### Option E: Apollo Client (full)
Pros: Native GraphQL caching with normalized store.
Cons: Heavy (~80KB), complex cache management, conflicts with App Router server components, slow cold start.

## Consequences

### Positive
- TanStack Query's `queryClient` enables cross-component cache invalidation (e.g., after fantasy transfer, invalidate fantasy team query everywhere)
- Zustand is tiny (~1KB), no context providers needed, works in any component
- Clear mental model: "is this data from the server?" → TanStack Query. "is this ephemeral browser state?" → Zustand
- TanStack Query v5 supports React 19 concurrent features used in Next.js 15

### Negative
- Developers (agents) must remember which library to use for which purpose — mitigated by clear conventions below

### Risks
- Stale UI after mutations: mitigated by `queryClient.invalidateQueries` after every mutation
- Zustand state lost on page refresh: by design — Zustand is for ephemeral state. Persistent state goes to server.

## Implementation Notes

**TanStack Query setup:**
```typescript
// apps/web/src/lib/query-client.ts
import { QueryClient } from '@tanstack/react-query';

export const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      staleTime: 60 * 1000, // 1 minute
      retry: 2,
    },
  },
});
```

**Query key conventions:**
```typescript
// Hierarchical keys enable targeted invalidation
export const queryKeys = {
  fantasy: {
    squad: (userId: string) => ['fantasy', 'squad', userId] as const,
    gameweek: (gameweekId: string) => ['fantasy', 'gameweek', gameweekId] as const,
  },
  football: {
    fixtures: (competitionId: string) => ['football', 'fixtures', competitionId] as const,
    standings: (seasonId: string) => ['football', 'standings', seasonId] as const,
  },
  loyalty: {
    account: (userId: string) => ['loyalty', 'account', userId] as const,
  },
};
```

**Zustand store conventions:**
```typescript
// One slice per domain — no mega-store
// apps/web/src/stores/ui.store.ts
interface UIState {
  transferModalOpen: boolean;
  transferModalPlayerId: string | null;
  openTransferModal: (playerId: string) => void;
  closeTransferModal: () => void;
}

export const useUIStore = create<UIState>((set) => ({
  transferModalOpen: false,
  transferModalPlayerId: null,
  openTransferModal: (playerId) => set({ transferModalOpen: true, transferModalPlayerId: playerId }),
  closeTransferModal: () => set({ transferModalOpen: false, transferModalPlayerId: null }),
}));
```

**Apollo Client (subscriptions only):**
```typescript
// Only used for live match subscriptions, not for queries
const wsLink = new GraphQLWsLink(createClient({
  url: process.env.NEXT_PUBLIC_WS_URL,
}));
```

**Server component + TanStack Query hydration pattern:**
```typescript
// Page component (server) prefetches, client component uses cache
// apps/web/src/app/fixtures/page.tsx
export default async function FixturesPage() {
  const queryClient = new QueryClient();
  await queryClient.prefetchQuery({
    queryKey: queryKeys.football.fixtures('psl'),
    queryFn: () => getFixtures('psl'),
  });
  return (
    <HydrationBoundary state={dehydrate(queryClient)}>
      <FixtureList />
    </HydrationBoundary>
  );
}
```

**Rule: No business logic in frontend.**
The frontend must NEVER:
- Calculate fantasy scores
- Validate squad composition rules (server-side only, see ADR-001/fantasy agent)
- Determine GTS prediction validity
- Compute loyalty tier

Frontend may ONLY:
- Display data returned by the API
- Provide UI affordances that prevent obviously invalid input (e.g., disable submit if form incomplete)
- Cache and display previously fetched server data
