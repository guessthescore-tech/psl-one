# ADR-018: Fantasy Rules Config-Driven

**Date:** 2026-06-14  
**Status:** ACCEPTED  
**Story:** STORY-14  

---

## Context

Early Fantasy scoring had hardcoded constants (points per goal, transfers per week, budget). These needed to vary between competitions (World Cup rules differ from PSL rules) and be tunable by admins without a code deployment.

## Decision

All Fantasy rules are stored in `FantasyRulesConfig` per season. Services read from this config at runtime. No scoring constants are hardcoded.

`FantasyRulesConfig` is managed via admin API at `POST /admin/fantasy-rules/:seasonId`.

`assertFantasyOpen()` reads deadline from `FantasyRulesConfig.transferDeadline` (via Gameweek) — not hardcoded.

## Alternatives Considered

- **Hardcoded constants per competition** — rejected; requires code deployment to tune rules, and rules differ per season
- **Rules in environment variables** — rejected; difficult to version per season, not auditable, and not admin-accessible at runtime
- **GraphQL schema-driven config** — deferred; overkill for a flat config structure

## Consequences

- Scoring is fully configurable per season
- Testing must inject a mock `FantasyRulesConfig` — tests don't hardcode point values
- A missing `FantasyRulesConfig` for a season causes Fantasy operations to fail — readiness check 4 enforces this
- PSL and World Cup can have different rules simultaneously

## Revisit Triggers

- If rule changes need to take effect mid-season (would require effective-date versioning on `FantasyRulesConfig`)
- If non-admin staff need self-service rule editing (add role-based config edit permission)
