# ADR-002: Monorepo Strategy

**Date:** 2026-06-08
**Status:** ACCEPTED
**Owner:** Platform Agent
**Reviewed By:** Programme Director Agent, Technical Review Board

## Context

PSL One consists of 15+ backend services, 4 frontend apps and 8+ shared packages. All are TypeScript. They share types, event schemas, auth guards, UI components, logging and testing utilities. Without a monorepo, keeping shared code in sync across separate repositories adds coordination overhead that would consume a significant fraction of the 12-week delivery window.

Requirements:
- Single repository for all services, apps, packages
- Affected-only CI (only build/test what changed)
- Shared packages importable by all services and apps
- pnpm for fast, reliable installs
- Node 22 LTS
- Works with GitHub Actions

## Decision

**Use Turborepo + pnpm workspaces.**

Package manager: `pnpm@9+`
Node version: `22 LTS`
Monorepo tool: `turbo@2+`

## Options Considered

### Option A: Turborepo + pnpm workspaces
Pros: Simple config, excellent build caching (local + remote via Vercel), TypeScript-first, `--filter` for affected-only builds, active community, Vercel-backed.
Cons: Fewer generators than NX; less opinionated (acceptable for this codebase).

### Option B: NX
Pros: Powerful code generators, affected graph, large plugin ecosystem, nx cloud caching.
Cons: More complex configuration, steeper learning curve, NX-specific conventions add friction for AI agents writing code.

### Option C: Lerna + pnpm
Pros: Flexible, well-understood.
Cons: Less intelligent caching, more manual pipeline configuration, less active development.

### Option D: Separate repositories per service
Pros: Maximum isolation, independent deployment cadence.
Cons: Shared-types hell, coordination overhead, cross-service refactors become multi-repo PRs — incompatible with 12-week timeline.

## Consequences

### Positive
- Single `pnpm install` installs all dependencies
- `turbo run build --filter=services/identity...` builds only identity and its dependencies
- Remote cache (Vercel Turborepo) makes CI fast after first run
- Shared packages (`packages/event-schemas`, `packages/shared-types`) imported with workspace aliases
- All agents work in one repository, reducing context-switching

### Negative
- Single repository means large git history over time
- All services must use compatible TypeScript/Node versions (acceptable — all are Node 22 + TS 5.x)

### Risks
- Large `node_modules` at root: mitigated by pnpm's content-addressed store (no duplication)

## Implementation Notes

**Root `package.json`:**
```json
{
  "name": "psl-one",
  "private": true,
  "packageManager": "pnpm@9.0.0",
  "engines": { "node": ">=22.0.0", "pnpm": ">=9.0.0" }
}
```

**`pnpm-workspace.yaml`:**
```yaml
packages:
  - 'apps/*'
  - 'services/*'
  - 'packages/*'
```

**`turbo.json` pipeline:**
```json
{
  "$schema": "https://turbo.build/schema.json",
  "tasks": {
    "build": { "dependsOn": ["^build"], "outputs": ["dist/**"] },
    "test": { "dependsOn": ["^build"] },
    "test:e2e": { "dependsOn": ["^build"] },
    "lint": {},
    "typecheck": { "dependsOn": ["^build"] },
    "dev": { "cache": false, "persistent": true }
  }
}
```

**Workspace package naming convention:**
- `@psl-one/shared-types` → `packages/shared-types`
- `@psl-one/event-schemas` → `packages/event-schemas`
- `@psl-one/identity` → `services/identity`
- `@psl-one/web` → `apps/web`

**`.nvmrc`:** `22`

**CI affected-only strategy:**
```yaml
- run: pnpm turbo run build test --filter=...[origin/main]
```
