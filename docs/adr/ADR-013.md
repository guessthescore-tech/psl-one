# ADR-013: Season Activation Gate (13-Check Readiness)

**Date:** 2026-06-14  
**Status:** ACCEPTED  
**Story:** STORY-28 (initial), STORY-36 (expanded to 13 checks)  

---

## Context

Switching the active season from World Cup to PSL carries significant risk. Fan data, Fantasy teams, predictions, and leaderboards are all scoped to seasons. An accidental activation could confuse fans and corrupt in-progress gameplay.

## Decision

Season activation requires 13 named readiness checks, all of which must pass before the switch is allowed. The checks are defined in `SeasonSwitchingService` and are the single source of truth — no other module duplicates them.

The 13 checks cover: clubs, published fixtures, gameweeks, fantasy rules, prediction rules, squad import, player prices, prediction calibration, gameweek ops, leaderboards, player stats, media campaigns, and beta cohort.

## Alternatives Considered

- **Single boolean `isReady` flag** — rejected; too coarse — operators need to know *which* check failed
- **Checks distributed across modules** — rejected; creates split ownership and risks checks being bypassed by one module calling another's internal state
- **Manual checklist in a runbook** — rejected; human checklists drift from code reality; automated checks are authoritative

## Consequences

- `SeasonSwitchingService` is the authoritative gatekeeper
- `BetaLaunchModule` delegates to it — no duplication (ADR-025)
- Adding a new readiness check requires updating `SeasonSwitchingService` only
- A check failure gives a clear error message identifying which check blocked
- PSL season is not activated in STORY-39 — approval record only (ADR-026)
