# ADR-010: Security & POPIA Strategy

**Date:** 2026-06-08
**Status:** ACCEPTED
**Owner:** Platform Agent (Identity + Security)
**Reviewed By:** Programme Director Agent, Technical Review Board, Security Review Agent

## Context

PSL One processes personal data of South African citizens, making it subject to POPIA (Protection of Personal Information Act). The platform handles authentication, financial transactions (loyalty points), and competitive game integrity. A security breach or POPIA violation carries legal liability, reputational damage, and potential fines of up to R10 million.

CLAUDE.md mandates: "Never bypass RBAC. Never bypass audit logs."

POPIA requires a legal review by a qualified practitioner before go-live. This ADR establishes the technical implementation decisions. Legal advice must be obtained separately.

## Decision

### Security Architecture

| Area | Decision |
|---|---|
| Auth tokens | JWT RS256. Public key distributed via JWKS endpoint. All services validate locally. |
| Access token lifetime | 15 minutes |
| Refresh token lifetime | 30 days. Stored httpOnly cookie. |
| Password hashing | bcrypt, cost factor 12 |
| Auth rate limiting | 10 requests per 15 minutes per IP for all auth endpoints |
| General rate limiting | 100 requests per minute per authenticated user |
| Input validation | Zod schemas at all API boundaries + class-validator in NestJS DTOs |
| SQL injection | Prisma parameterised queries (automatic protection). No raw SQL allowed. |
| XSS | CSP headers, `Content-Security-Policy: default-src 'self'`, no `dangerouslySetInnerHTML` |
| CSRF | `SameSite=Strict` cookies for auth tokens, CSRF token for state-changing operations from web UI |
| Secrets management | AWS Secrets Manager only. No secrets in environment variables in production. No secrets in git. |
| TLS | TLS 1.3 minimum. HSTS enabled. Enforced by CloudFront + ALB. |
| Dependency scanning | GitHub Dependabot weekly. No known critical/high CVEs in production. |
| Container scanning | AWS ECR image scanning on push. Block deployment on critical CVEs. |

### RBAC Roles

| Role | Description | Permissions |
|---|---|---|
| `FAN` | Registered fan | Own data, fantasy, GTS, loyalty read/write |
| `CLUB_ADMIN` | Club staff | Club content, club players, club roster |
| `SPONSOR_ADMIN` | Sponsor staff | Own campaign data, aggregated fan segments (no PII) |
| `PSL_ADMIN` | PSL staff | All football data, content, all clubs |
| `SUPER_ADMIN` | Platform operators | Everything except audit log deletion |
| `COMPLIANCE_OFFICER` | POPIA/legal officer | User data access, consent records, erasure requests |

### POPIA Compliance

| POPIA Requirement | Implementation |
|---|---|
| Lawful basis | Explicit consent at registration, per purpose |
| Consent capture | At registration: MARKETING, ANALYTICS, THIRD_PARTY_SHARING shown separately with clear language |
| Consent storage | Immutable `consent_records` table — append-only, no UPDATE or DELETE |
| Right to access | `GET /api/v1/me/data` — returns all personal data; fulfilled within 24 hours |
| Right to erasure | `DELETE /api/v1/me/account` — anonymises PII within 30 days, retains anonymised analytics |
| Data minimisation | Collect only: name, email, mobile, province, club. No ID number. No biometrics. |
| Data retention | Active users: indefinite. Deleted users: anonymised after 30 days. Logs: 90 days. |
| Breach notification | POPIA requires notification to Information Regulator within 72 hours. Runbook required. |
| Data portability | `GET /api/v1/me/data/export` — JSON export within 24 hours |
| Minor protection | Age gate at registration. Under-18 flag. Guardian consent mechanism (Phase 2). |
| Information Officer | Must be designated by PSL before go-live. Details required in privacy policy. |

## Options Considered

### Token Algorithm
- RS256 (asymmetric, public key verification): CHOSEN — services verify without calling auth service, public key safely distributable
- HS256 (symmetric, shared secret): Rejected — all services would share a secret; one compromise = all services compromised

### Password Hashing
- bcrypt cost 12: CHOSEN — reasonable security/performance balance (~300ms on modern hardware)
- bcrypt cost 10: Rejected — too fast for brute-force resistance
- Argon2id: Future consideration for Phase 2 — bcrypt is sufficient for Phase 1

### Consent Model
- Granular per-purpose consent (MARKETING, ANALYTICS, THIRD_PARTY_SHARING): CHOSEN — POPIA requires granular consent, not blanket opt-in
- Single "I agree" checkbox: Rejected — insufficient under POPIA

## Consequences

### Positive
- JWT RS256 means auth service is not a bottleneck — each service validates tokens independently
- Immutable `consent_records` table means consent history is auditable and legally defensible
- bcrypt cost 12 makes brute-force attacks computationally expensive
- POPIA endpoints built into Identity Service from day one — no retrofit required

### Negative
- 15-minute access token expiry means more frequent refresh token calls (acceptable UX trade-off for security)
- Granular consent adds one extra step to registration (acceptable — required by law)

### Risks
- Stolen refresh token: mitigated by `SameSite=Strict` httpOnly cookie, refresh token rotation
- POPIA consent model found inadequate by legal review: mitigated by legal review in Sprint 0 (before code is written)
- JWT RS256 key rotation: mitigated by JWKS endpoint allowing key rotation without service restarts

## Implementation Notes

**JWT validation in NestJS services:**
```typescript
// packages/auth-guards/src/jwt.guard.ts
import { CognitoJwtVerifier } from 'aws-jwt-verify';

@Injectable()
export class JwtAuthGuard implements CanActivate {
  private readonly verifier = CognitoJwtVerifier.create({
    userPoolId: process.env.COGNITO_USER_POOL_ID,
    tokenUse: 'access',
    clientId: process.env.COGNITO_CLIENT_ID,
  });

  async canActivate(context: ExecutionContext): Promise<boolean> {
    const token = this.extractToken(context);
    const payload = await this.verifier.verify(token); // validates RS256 signature locally
    request.user = payload;
    return true;
  }
}
```

**RBAC guard:**
```typescript
// packages/auth-guards/src/roles.guard.ts
@Injectable()
export class RolesGuard implements CanActivate {
  canActivate(context: ExecutionContext): boolean {
    const requiredRoles = this.reflector.get<Role[]>('roles', context.getHandler());
    const { user } = context.switchToHttp().getRequest();
    const userRoles: Role[] = user['cognito:groups'] ?? [];
    return requiredRoles.some(role => userRoles.includes(role));
  }
}

// Usage:
@Roles(Role.PSL_ADMIN, Role.SUPER_ADMIN)
@UseGuards(JwtAuthGuard, RolesGuard)
@Post('/admin/fixtures')
createFixture() { ... }
```

**Consent record table (Identity service Prisma schema):**
```prisma
model ConsentRecord {
  id        String   @id @default(uuid())
  userId    String   @map("user_id")
  purpose   ConsentPurpose
  granted   Boolean
  version   String   @default("1.0")
  ipAddress String   @map("ip_address")
  userAgent String?  @map("user_agent")
  createdAt DateTime @default(now()) @map("created_at")
  // NO updatedAt — this table is append-only
  
  @@map("consent_records")
  @@index([userId])
}

enum ConsentPurpose {
  MARKETING
  ANALYTICS
  THIRD_PARTY_SHARING
}
```

**POPIA endpoint implementation (Identity Service):**
```
GET  /api/v1/me/data         — aggregate all personal data across services (async, 24h fulfillment)
GET  /api/v1/me/data/export  — JSON export download
POST /api/v1/me/consent      — update consent preferences (creates new ConsentRecord row)
GET  /api/v1/me/consent      — view current consent state
DELETE /api/v1/me/account    — initiate account deletion (30-day anonymisation)
```

**Secrets management rules:**
- All secrets stored in AWS Secrets Manager
- ECS tasks access secrets via IAM task role (no env vars in task definitions for secrets)
- Secret rotation: database passwords rotate every 30 days automatically
- `git-secrets` pre-commit hook blocks committing AWS keys, passwords, connection strings

**POPIA legal review — required before Sprint 1:**
A qualified POPIA practitioner must review and sign off on:
1. The consent capture wording at registration
2. The `consent_records` data model
3. The right-to-erasure anonymisation approach
4. The data retention policy
5. The breach notification runbook
6. The Information Officer designation
