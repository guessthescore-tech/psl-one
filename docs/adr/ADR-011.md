# ADR-011: Bootstrap MVP Architecture

**Date:** 2026-06-08  
**Status:** ACCEPTED — supersedes ADR-003, ADR-004, ADR-005, ADR-006 for Phase 1  
**Authority:** Architecture Review Board  
**Review trigger:** Funding secured OR 50K MAU reached OR infrastructure cost exceeds $100/month

---

## Context

The ARB review (ARB-001) identified that the original architecture's infrastructure cost exceeds Year 1 revenue. The programme requires a working, demonstrable, user-testable platform before external funding is secured.

This ADR defines a **bootstrap MVP architecture** that:
- Runs within AWS Free Tier / $100/month budget
- Preserves all domain boundaries for clean extraction later
- Meets all security and POPIA requirements
- Produces a platform capable of onboarding real users, real data, and a fundraising demo

The original ADRs (001–010) remain the target state. This ADR defines the path to get there.

## Rejected Alternatives

### Option A: Proceed with microservices architecture (ADR-001–010 as written)

**Rejected because:** Infrastructure cost ~$4,785/month across environments. Year 1 revenue R2M. Before a single user, the team spends 30% of annual revenue on infrastructure.

### Option B: Use non-AWS managed services (Railway, Render, Supabase)

**Rejected because:** Moves away from the AWS-native target architecture. Migration back to AWS costs rework time. AWS free tier provides equivalent capability at $0.

### Option C: Static-only MVP (no backend, no database)

**Rejected because:** Cannot validate the core product mechanics (GTS, fantasy scoring, loyalty). A static demo does not validate the platform.

## Component Decisions

### 1. Backend: NestJS Modular Monolith

**Decision:** One NestJS application (`services/api/`) containing all bounded contexts as NestJS modules.

**Module structure:**
```
services/api/src/modules/
  identity/      ← BoundedContext: auth, users, POPIA
  fan/           ← BoundedContext: profiles, preferences
  football/      ← BoundedContext: fixtures, clubs, players
  fantasy/       ← BoundedContext: squads, scoring
  gts/           ← BoundedContext: predictions, settlement
  loyalty/       ← BoundedContext: points, tiers
  wallet/        ← BoundedContext: balance, transactions
  content/       ← BoundedContext: articles, media
  notifications/ ← BoundedContext: email, push delivery
  admin/         ← BoundedContext: admin overrides
  outbox/        ← Infrastructure: outbox worker
  health/        ← Infrastructure: health endpoints
```

**Domain boundary rule:** Modules MUST NOT import each other's `Service` classes. Cross-module communication happens exclusively through:
1. `EventEmitter2` (in-process, async, non-blocking) — for domain events
2. Module method calls via an interface (synchronous, for queries only)

This enforces the same boundary contract as inter-service HTTP/Kafka. When we extract a module to its own service, the EventEmitter2 calls become Kafka produces with no logic changes.

**Rationale:** Monolith allows the team to ship features without managing inter-service HTTP, distributed transactions, or network partitions. The boundary discipline remains identical to the target microservice architecture.

---

### 2. API Layer: GraphQL (Unified, No Federation)

**Decision:** Single unified GraphQL schema via `@nestjs/graphql` code-first. No Apollo Federation directives for Phase 1.

**Why not REST:** The frontend is designed to use GraphQL. Changing to REST and back creates rework.

**Why not Federation:** Federation requires multiple running services. In a monolith, one unified schema is equivalent and simpler.

**Migration:** When modules are extracted to services, add `@key`, `@external`, and `@provides` directives to the SDL. Apollo Router sits in front. The frontend sees no change.

**Gateway:** During bootstrap, the NestJS app IS the gateway. No Apollo Router needed.

---

### 3. Database: PostgreSQL Multi-Schema (Single RDS Instance)

**Decision:** One AWS RDS db.t3.micro PostgreSQL instance. Each domain has its own PostgreSQL schema. Prisma multi-schema feature (`multiSchema` preview).

**Schema layout:**
```sql
CREATE SCHEMA identity;    -- users, consent_records, otp_tokens
CREATE SCHEMA fan;         -- profiles, preferences, club_affiliations
CREATE SCHEMA football;    -- competitions, seasons, fixtures, clubs, players
CREATE SCHEMA fantasy;     -- squads, picks, gameweeks, scores
CREATE SCHEMA gts;         -- predictions, settlements
CREATE SCHEMA loyalty;     -- accounts, transactions, tiers
CREATE SCHEMA wallet;      -- accounts, ledger
CREATE SCHEMA content;     -- articles, media
CREATE SCHEMA notifications; -- templates, preferences, deliveries
CREATE SCHEMA outbox;      -- outbox_events (shared, infrastructure)
CREATE SCHEMA audit;       -- audit_log (shared, infrastructure)
```

**Cross-schema JOIN policy:** PROHIBITED. No query may JOIN across domain schemas. If module A needs data from module B, it calls module B's service method or reads from a denormalised read model within its own schema. This is the same rule as cross-service DB access in the microservice architecture — enforced here by convention + code review.

**Prisma configuration:**
```prisma
generator client {
  provider        = "prisma-client-js"
  previewFeatures = ["multiSchema"]
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
  schemas  = ["identity", "fan", "football", "fantasy", "gts", 
               "loyalty", "wallet", "content", "notifications",
               "outbox", "audit"]
}
```

**Why not one schema per database:** With 11 databases on a `db.t3.micro`, connection overhead is unmanageable and the free tier limit (20GB total) becomes complex to manage. Logical schemas give the same isolation at a fraction of the cost.

**Migration path:** When a module is extracted to its own service, `pg_dump --schema=<domain>` exports the schema to a new RDS instance. Connection string changes in the extracted service. Data is migrated via pg_restore. Zero application logic changes.

### 4. Events: PostgreSQL Outbox + NestJS EventEmitter2

**Decision:** No Kafka. No MSK. No SQS for Phase 1.

**Event flow:**
```
Domain service method called
  ↓
Prisma transaction:
  1. Write domain entity change
  2. Write OutboxEvent row (topic, payload, status=PENDING)
  ↓
Transaction commits
  ↓
EventEmitter2.emit(topic, payload)  ← in-process, immediate
  ↓
Event handlers in other modules execute synchronously (within request)
OR
  ↓
OutboxWorker polls outbox_events every 5s
  ↓
Dispatches events to registered in-process handlers
  ↓
Marks OutboxEvent as PUBLISHED
```

**OutboxEvent schema:**
```sql
-- outbox.outbox_events
id            UUID PRIMARY KEY DEFAULT gen_random_uuid()
topic         VARCHAR NOT NULL   -- e.g. identity.user.registered
payload       JSONB NOT NULL     -- KafkaEventEnvelope shape
status        ENUM (PENDING, PUBLISHED, FAILED, DEAD)
attempts      INT DEFAULT 0
created_at    TIMESTAMPTZ DEFAULT NOW()
published_at  TIMESTAMPTZ
error_message TEXT
```

**Audit trail:** Every domain state change writes an OutboxEvent. This means the outbox IS the event log. When Kafka is introduced post-funding, the OutboxWorker changes from "dispatch in-process" to "produce to Kafka." No application logic changes.

**Why preserve the OutboxEvent write:** If we write events in-process only (EventEmitter2 without the DB write), we have no audit trail, no replay capability, and no migration path to Kafka. The DB write is the foundation.

**Redis:** Not needed in Phase 1. Rate limiting uses NestJS `@nestjs/throttler` with in-memory store. Live score updates use HTTP polling (5s interval) rather than WebSocket subscriptions for MVP. WebSockets are added when the subscription architecture is designed properly (ADR-003 finding 003-C).

### 5. Authentication: AWS Cognito (Simplified Flow)

**Decision:** AWS Cognito remains the auth provider. Free tier covers 50K MAU — sufficient for MVP.

**Simplification from ADR-001:** Remove the Lambda Post-Confirmation trigger saga. Instead:

```
Fan submits registration form
  ↓
Frontend calls POST /api/v1/auth/register (NestJS)
  ↓
NestJS: adminCreateUser in Cognito (server-side)
  ↓
Prisma transaction:
  1. identity.users row
  2. identity.consent_records row
  3. outbox.outbox_events row (identity.user.registered)
  ↓
EventEmitter2 emits USER_REGISTERED
  ↓
fan module creates profile
loyalty module creates account
wallet module creates wallet
  ↓
Return success to frontend
  ↓
Cognito sends verification email
```

**Why this is better than Lambda trigger:** The registration logic runs in the NestJS monolith where it is testable, observable, and transactional. No distributed transaction across Cognito → Lambda → NestJS. If any step fails, the Prisma transaction rolls back and the Cognito user is not created.

**JWT validation:** `aws-jwt-verify` in the NestJS auth guard. Same as ADR-001. No change to the downstream auth flow.

**Cognito exit strategy:** The `registerWithCognito()` call is isolated in `identity/adapters/cognito.adapter.ts` behind a `IAuthProvider` interface. Future auth providers (Supabase Auth, Clerk) swap the adapter only.

### 6. Frontend: Vercel (Free Tier)

**Decision:** Deploy `apps/web` to Vercel. Use Vercel Hobby plan (free, unlimited deployments).

**Why Vercel over Amplify:**
- Vercel is built by the Next.js team. Zero configuration for Next.js 15.
- Preview deployments per PR — essential for AI agent workflow
- Edge network, image optimization, ISR all included free
- Amplify SSR support for Next.js is more complex and has known limitations

**Domain:** Connect `pslone.co.za` to Vercel production deployment. CNAME managed by Vercel.

**API calls:** `NEXT_PUBLIC_API_URL=https://api.pslone.co.za` — calls the NestJS monolith on EC2.

**Migration:** If moving off Vercel post-funding (e.g., to Amplify or self-hosted CloudFront), the Next.js app is unchanged. Only the deployment target and CI config changes.

---

### 7. Infrastructure: EC2 t2.micro + Docker Compose

**Decision:** NestJS monolith runs in Docker on a single EC2 t2.micro. Nginx as reverse proxy. Let's Encrypt for TLS.

**Why not ECS Fargate:** ECS Fargate costs ~$30-50/month minimum. EC2 t2.micro free tier = $0. For a single-container application, Docker Compose on EC2 is operationally equivalent. ECS Fargate is added when the first service is extracted from the monolith.

**Why not serverless (Lambda):** NestJS is not designed for Lambda cold starts. The monolith has 100ms+ startup time. Lambda requires architectural changes (e.g., `serverless-express` wrapper) that complicate the migration to ECS later.

**Docker Compose on EC2 (production):**
```yaml
services:
  nginx:
    image: nginx:alpine
    ports: ["80:80", "443:443"]
    volumes: [certbot certs, nginx config]
    
  api:
    image: <ecr-repo>/psl-api:latest
    environment: [DATABASE_URL, COGNITO_*, etc.]
    restart: unless-stopped
```

**Deployment:** GitHub Actions pushes Docker image to ECR, SSHes to EC2, runs `docker compose pull && docker compose up -d`. Zero-downtime restart via Nginx upstream with health check.

**Database:** AWS RDS PostgreSQL db.t3.micro in the same VPC as the EC2 instance. Private subnet (no public IP). EC2 connects via private IP.

### 8. Terraform Scope (Bootstrap)

**Decision:** Minimal Terraform. One file per environment concept. No modules yet.

```
infra/terraform/bootstrap/
  main.tf          ← VPC (default), security groups, EC2, RDS, S3, IAM
  variables.tf     ← region, environment, db_password (from env)
  outputs.tf       ← ec2_ip, rds_endpoint, s3_bucket
  .gitignore       ← *.tfstate*, .terraform/
```

**State:** Local `terraform.tfstate` during development (added to `.gitignore`). Move to S3 backend when the first non-dev environment is needed.

**What Terraform does NOT manage yet:** Cognito (configured manually or via AWS CLI), CloudFront (optional for now), WAF (added with first public load test).

### 9. What We Give Up (and Why It's Acceptable)

| Capability | Bootstrap Approach | Acceptable Because |
|---|---|---|
| Horizontal scaling | None (single EC2 t2.micro) | MVP has <1K users. No concurrent load. |
| Service independence | Monolith deployment | All agents still work in bounded modules |
| Independent service upgrades | Whole-app redeploy | Fast: < 30 second deploy |
| Kafka event streaming | In-process + DB outbox | Outbox maintains the audit trail and migration path |
| Multi-AZ HA | Single instance | MVP downtime is acceptable during off-hours |
| Per-service databases | Logical schemas | Identical isolation rules, lower cost |
| Live WebSocket scores | HTTP polling (5s) | Acceptable UX for MVP demo |
| Redis caching | No cache | Performance acceptable at <1K users |
| ECS Fargate | Docker on EC2 | Identical containers, different orchestrator |

### 10. Migration Triggers

This architecture MUST be revisited when ANY of the following occur:

| Trigger | Action Required |
|---|---|
| Funding secured (any amount) | Begin Phase 2 migration (see `docs/architecture/migration-path.md`) |
| EC2 free tier expires (12 months) | Move to ECS Fargate OR reserved EC2 ($8/month) |
| > 10K concurrent users | Extract Football + GTS to separate services |
| > 50K MAU | Cognito cost assessment, consider auth consolidation |
| Monolith deploy time > 5 minutes | Extract first service (probably Football Data) |
| Any single module > 50K lines of code | Extract that module |
| Database size > 15GB | Aurora migration (before hitting 20GB RDS free tier) |

## Consequences

**Positive:**
- Infrastructure cost: $0/month (free tier), $29/month after
- All domain boundaries preserved for extraction
- Full POPIA and security compliance maintained
- Deployable in Sprint 0 (1 week vs 2 weeks for microservices)
- Faster feature development (no network calls between modules)
- Full audit trail via outbox_events table

**Negative:**
- No horizontal scaling (mitigated: not needed at MVP scale)
- All-or-nothing deployment (mitigated: fast deploys, rollback easy)
- Single database is a performance bottleneck at scale (mitigated: extraction path defined)
- Vercel vendor dependency for frontend (mitigated: Next.js is portable)

**Risks accepted:**
- EC2 t2.micro failure = full outage (no HA). Acceptable for MVP. Alert on instance health.
- RDS single-AZ failure = full outage. RDS automated backups enabled. RPO: last backup (24h).
