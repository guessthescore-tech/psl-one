# ADR-023: getBetaToken Centralisation

**Date:** 2026-06-14  
**Status:** ACCEPTED  
**Story:** STORY-35  

---

## Context

Test files across 54+ spec files each needed a way to generate a valid JWT for test scenarios. Early test files copy-pasted a local `getToken()` helper. As JWT structure evolved (role field, expiry), each file needed to be updated independently, leading to drift and test failures.

## Decision

A single `getBetaToken()` function is exported from `apps/api/src/auth/test-helpers.ts`. All spec files import from this single location.

The function signature:
```typescript
getBetaToken({ id, email, role }: { id: string; email?: string; role?: string }): string
```

## Alternatives Considered

- **Shared test fixture factory** — considered; a factory that creates a full User in the DB is more realistic but slower and couples test setup to DB migrations; `getBetaToken()` is faster and isolated
- **Per-spec token helpers** — the existing broken state before this ADR; copy-pasted helpers drifted; rejected in favour of single source
- **NestJS `@nestjs/passport` mock** — more complex than needed; JWT generation is the only variation point between tests

## Consequences

- One change to the JWT structure requires one file update
- Test setup is consistent across all spec files
- Reviewers can spot test auth issues by looking at one file
- No spec file should define its own token-generation helper
- If a test needs a custom claim, extend `getBetaToken()` with an optional parameter

## Related Stories

STORY-35 (centralisation fix), applicable to all 54+ spec files

## Related Source Files

- `apps/api/src/auth/test-helpers.ts` — exports `getBetaToken()`
- All `*.spec.ts` files — import from `test-helpers.ts`

## Revisit Triggers

- If JWT claims structure changes significantly (update `getBetaToken()` signature, not individual specs)
- If E2E tests require a real signed JWT from a test user (then add `createTestUser()` to test-helpers alongside `getBetaToken()`)
