# ADR-022: No Business Logic in Frontend

**Date:** 2026-06-14  
**Status:** ACCEPTED  
**Project rule — all stories**  

---

## Context

In Next.js App Router with client components, it is tempting to add validation logic in the browser (e.g., checking if a transfer window is open, calculating budget remaining). This creates two copies of the same logic with divergence risk.

## Decision

All business logic lives in the NestJS API. The frontend:
- Displays data returned by the API
- Submits requests to the API
- Shows the API's error messages
- Never validates domain rules independently

Role-based UI hints (show/hide admin links) are acceptable in the frontend, but the API always enforces the actual access control.

## Alternatives Considered

- **Shared validation library (zod schema re-use)** — considered; could share validation types between API and web but still puts validation logic in both places; rejected to keep the boundary clean
- **GraphQL with client-side schema** — the planned federation approach (ADR-011) will carry business logic server-side regardless of transport; this ADR reinforces the principle for the REST MVP
- **BFF (Backend for Frontend)** — deferred; not needed at current scale; Next.js server components serve the same purpose for SSR data fetching

## Consequences

- Every mutation goes through the API — no client-side optimistic mutations without an API call
- API is the single source of truth for all domain state
- Frontend can be completely replaced without changing domain rules
- Doubles the API call count vs. pure client-side UX — acceptable for beta
- `NEXT_PUBLIC_API_BASE_URL` is the only required web environment variable (`http://localhost:4000` default)

All stories — enforced as a project rule from STORY-01 onwards

## Related Source Files

- `apps/web/src/lib/*-client.ts` — all client functions call the API; none compute domain state

## Revisit Triggers

- If a BFF layer is introduced for SSR performance (Next.js server components can call API directly — still no browser logic)
- If offline support is required (then client-side state would need a sync protocol — new ADR needed)
