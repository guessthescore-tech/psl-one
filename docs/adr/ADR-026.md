# ADR-026: PSL Season Activation Not Implemented in STORY-39

**Date:** 2026-06-14  
**Status:** ACCEPTED  
**Story:** STORY-39  

---

## Context

STORY-39 (Beta Launch Readiness) delivers cohort management and an activation approval record (`SeasonActivationApproval`). There was pressure to also include the actual season activation endpoint as part of the "launch readiness" story.

## Decision

PSL season activation is deliberately not implemented in STORY-39 or any prior story. The reasons:

1. **Official PSL data not received** — squad import check (check 6) requires official PSL 2026/27 player registrations, which have not been provided (STORY-40 dependency)
2. **Activation is a separate operational milestone** — it should be a named, audited, explicit operation with its own story
3. **Safety isolation** — any activation-related code exists only in `SeasonSwitchingService`, which is already written. Adding an activation API endpoint is trivial when the time comes

## Acceptance Criteria

The following hold true in all code shipped through STORY-39:

- `Season.isActive` is never changed by `BetaLaunchModule`
- `SeasonActivationApproval.approvalStatus` is `APPROVED`, never `ACTIVATED`
- `activationPerformedAt` is never set
- No API route exists that triggers `SeasonSwitchingService.switchSeason()`
- `BetaLaunchSmokeTestService` verifies `activationRouteAbsent: true`
- The `ACTIVATION_DISABLED_NOTICE` constant is exported and included in relevant responses

## Alternatives Considered

- **Include activation in STORY-39** — rejected; official PSL player data is not yet available (STORY-40 dependency); activating without squad data would leave an incomplete live experience
- **Activation as a scheduled task** — rejected; PSL season launch must be an explicit, audited, human decision — not automated
- **Dual-active seasons** — not applicable; the architecture supports only one active season per competition at a time

## Consequences

- The PSL season remains inactive and must be activated explicitly in a future sprint
- STORY-40 must be completed first (official PSL data)
- Any engineer who adds a season activation API route is bypassing this explicit decision and must create a new ADR
- `BetaLaunchSmokeTestService` verifies `activationRouteAbsent: true` on every smoke test run
- The `ACTIVATION_DISABLED_NOTICE` constant is exported and included in relevant responses

## Related Stories

STORY-39 (BetaLaunchModule, SeasonActivationApproval), STORY-40 (planned — official PSL squad data, unblocks activation)

## Related Source Files

- `apps/api/src/beta-launch/beta-launch.service.ts` — exports `ACTIVATION_DISABLED_NOTICE`
- `apps/api/src/beta-launch/beta-launch-smoke-test.service.ts` — `activationRouteAbsent` check
- `apps/api/prisma/schema.prisma` — `SeasonActivationApproval` (`approvalStatus` never `ACTIVATED`)
- `apps/api/src/season-switching/season-switching.service.ts` — `switchSeason()` method (exists but not exposed via API)

## Revisit Triggers

- When STORY-40 is complete and official PSL squad data is loaded
- When product explicitly issues the instruction to activate the PSL season (requires new ADR or amendment of this one)
