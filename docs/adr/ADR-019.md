# ADR-019: Sandbox-First Provider Pattern

**Date:** 2026-06-14  
**Status:** ACCEPTED  
**Story:** STORY-37 (Wallet)  

---

## Context

PSL One integrates with external providers — wallet, notification, football data, CDN. Most of these require signed contracts that don't exist yet. Development cannot block on contract negotiations.

## Decision

All external integrations use an adapter pattern with a sandbox-first approach:

1. Define a TypeScript interface for the provider contract
2. Implement a sandbox adapter that returns mock data with zero outbound HTTP calls
3. The sandbox adapter is the default in all environments
4. A production adapter is implemented separately when the provider is contracted
5. Switch via environment variable: `WALLET_PROVIDER=production`

The `SiliconEnterpriseSandboxWalletAdapter` is the reference implementation.

## Alternatives Considered

- **Feature flags disabling integration calls** — rejected; doesn't enforce the interface contract, making it easy to skip without realising
- **Mock service workers (MSW)** — considered for tests but not for runtime adapter; MSW intercepts HTTP which is harder to control than an adapter swap
- **No integration until contracts signed** — rejected; would block all wallet, notification, and live-data development

## Consequences

- No real money can be moved in any environment until a production adapter is explicitly configured
- Development and testing never require external API keys
- `IntegrationProviderConfig` tracks which providers are configured and at what level
- All sandbox adapters must implement the full interface — no stubs or partial implementations
- Production adapter wiring requires a new ADR to document the provider decision
