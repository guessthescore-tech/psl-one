# ADR-007: Football Data Provider Strategy

**Date:** 2026-06-08
**Status:** ACCEPTED
**Owner:** Football Core Agent
**Reviewed By:** Programme Director Agent, Technical Review Board

## Context

PSL One requires reliable, timely football data: PSL and MTN8 fixtures, results, player statistics, and live match events. This data drives the Fantasy scoring engine, GTS settlement, the match centre, and league standings. Without real data, the platform has no content.

The choice of data provider has significant impact on:
- Fantasy scoring accuracy and timeliness
- GTS settlement reliability
- Match centre engagement value
- Year 1 infrastructure cost (R2M revenue target)

A human decision is required to contract a data provider before Sprint 1 begins. This ADR establishes the recommended direction and the technical implementation pattern regardless of which provider is chosen.

## Decision

**API-Football (via RapidAPI) for Phase 1 + mandatory manual override capability.**

Rationale: Budget-appropriate for Year 1 revenue. PSL fixture schedules are known in advance. Fantasy scoring tolerates 15-minute data delay (not live-scoring). The manual override admin API covers all data gaps.

Phase 2: Evaluate Sportradar when revenue and live-scoring requirements justify the cost.

**Critical technical decision (independent of provider):** The Football Service MUST implement a `FootballDataProviderPort` interface. The provider is an interchangeable adapter. Domain logic never calls the provider directly.

## Options Considered

| Provider | Coverage | Cost (est.) | Real-time | PSL Coverage | Decision |
|---|---|---|---|---|---|
| **API-Football (RapidAPI)** | Good | R500–2K/month | 15-min delay | Partial | CHOSEN for Phase 1 |
| Sportradar | Excellent | R15K–50K/month | <10s | Yes | Deferred to Phase 2 |
| Opta (Stats Perform) | Premium | R20K–80K/month | Yes | Yes | Deferred to Phase 2 |
| LiveScore API | Basic | R200–500/month | 15-min delay | Limited | Fallback option |
| Manual entry only | Full control | Staff cost | No | Full | Available as override |

## Consequences

### Positive
- Phase 1 cost is within Year 1 budget constraint
- Manual override endpoint covers all API-Football PSL data gaps
- Provider abstraction means upgrading to Sportradar in Phase 2 requires only a new adapter, not domain model changes

### Negative
- 15-minute data delay means fantasy scores are not real-time (acceptable for Phase 1 — delay matches expectations)
- API-Football PSL coverage may have gaps for player IDs and statistics
- Manual data entry creates staff operational burden

### Risks
- Provider API outage: mitigated by cached data + manual override admin endpoints
- PSL data quality issues: mitigated by admin manual override for all football data
- Provider contract cancellation: mitigated by adapter pattern (swap provider without domain changes)

## Implementation Notes

**Port/Adapter pattern (mandatory):**
```typescript
// services/football/src/provider/football-data-provider.port.ts
export interface FootballDataProviderPort {
  getFixtures(seasonId: string, from: Date, to: Date): Promise<FixtureDto[]>;
  getFixture(externalFixtureId: string): Promise<FixtureDto>;
  getLiveFixtures(competitionId: string): Promise<FixtureDto[]>;
  getStandings(seasonId: string): Promise<StandingDto[]>;
  getPlayerStats(fixtureId: string): Promise<PlayerStatDto[]>;
  getSquad(clubId: string, seasonId: string): Promise<PlayerDto[]>;
}
```

**API-Football adapter:**
```typescript
// services/football/src/provider/api-football/api-football.adapter.ts
@Injectable()
export class ApiFootballAdapter implements FootballDataProviderPort {
  constructor(
    private readonly httpService: HttpService,
    private readonly config: ConfigService,
  ) {}

  async getFixtures(seasonId: string, from: Date, to: Date): Promise<FixtureDto[]> {
    const response = await this.httpService.axiosRef.get('/fixtures', {
      headers: { 'x-rapidapi-key': this.config.get('API_FOOTBALL_KEY') },
      params: { league: this.mapSeasonToLeagueId(seasonId), from, to },
    });
    return response.data.response.map(this.mapFixture);
  }
  
  // ACL: maps external API shape to internal domain DTOs
  private mapFixture(raw: ApiFootballFixture): FixtureDto { ... }
}
```

**Manual override admin REST API (required regardless of provider):**
```
POST /admin/fixtures              — create fixture manually
PATCH /admin/fixtures/:id         — update fixture (score, status, events)
POST /admin/fixtures/:id/events   — add goal/card/substitution manually
POST /admin/standings/recalculate — trigger standings recalculation
POST /admin/player-stats/:fixtureId — override player statistics
```

**Data sync strategy:**
- Scheduled job every 15 minutes during match windows (ECS Scheduled Task)
- Manual trigger available from admin portal
- All external data written to `imported_fixtures` staging table, then reconciled to domain tables
- Reconciliation preserves any manual overrides (manual > provider)

**NestJS module wiring:**
```typescript
// FootballModule registers the adapter via injection token
providers: [
  {
    provide: 'FOOTBALL_DATA_PROVIDER',
    useClass: ApiFootballAdapter, // swap to SporradarAdapter in Phase 2
  },
  FootballSyncService,
]
```

- Contract API-Football account before Sprint 1 (Day 1)
- Store API key in AWS Secrets Manager: `psl/football/api-football-key`
- Verify PSL league ID in API-Football (test request before Sprint 1)
