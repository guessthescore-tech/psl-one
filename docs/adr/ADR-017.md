# ADR-017: Admin Audit Log Requirement

**Date:** 2026-06-14  
**Status:** ACCEPTED  
**Story:** STORY-35  

---

## Context

Admin users perform powerful mutations — settling predictions, scoring gameweeks, voiding challenges, switching seasons. Without an audit trail, there is no accountability and no ability to investigate disputes.

## Decision

Every admin mutation must write an `AdminAuditLog` record before returning. The actor's user ID is always extracted from the JWT context (`req.user.id`), never from the request body.

`AdminAuditLog` is immutable — rows are never updated or deleted.

## Alternatives Considered

- **Application-level change log** — less structured than a dedicated table; harder to query by actor or entity type
- **Database-level triggers** — rejected; bypasses NestJS and captures only schema-level changes, not intent (e.g., "settled gameweek X" vs. raw UPDATE)
- **External audit SaaS** — rejected for beta; adds cost and external dependency for a capability that can be built simply

## Consequences

- All admin service methods must accept an `adminUserId: string` parameter
- Tests must assert that `adminAuditLog.create` was called with the correct actor
- Schema migrations must never add `ON DELETE CASCADE` to `AdminAuditLog`
- "Never bypass audit logs" is an explicit project rule with no exceptions
- If an admin mutation fails, the audit log write is rolled back with it (transaction scope)
