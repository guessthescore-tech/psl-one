# ADR-021: Idempotency Key for Challenge Acceptance

**Date:** 2026-06-14  
**Status:** ACCEPTED  
**Story:** STORY-38  

---

## Context

A fan could double-tap an "Accept" button, or a network retry could send duplicate acceptance requests. Without idempotency, a fan could accept the same challenge twice, committing double the points.

## Decision

Challenge acceptance uses a `ChallengeIdempotency` record with a composite key:

- Marketplace: `marketplace-accept:{listingId}:{fanUserId}`
- Direct challenge: `direct-accept:{listingId}:{fanUserId}`

This key is written atomically in the same database transaction as the match creation and points commitment. If the key already exists, the transaction fails with a `ConflictException`.

## Alternatives Considered

- **Unique constraint on (listingId, fanUserId)** — considered; a DB constraint would enforce it but gives a cryptic Prisma error; the idempotency key gives a controlled `ConflictException` with a user-friendly message
- **Client-side disable after first tap** — rejected; network retries bypass client-side disabling; server-side enforcement is required regardless
- **Redis lock** — considered for high-throughput scenarios; not needed at beta scale; revisit if concurrent acceptance load becomes an issue

## Consequences

- Duplicate acceptance requests are safely rejected
- The rejection message is non-alarming — "Challenge already accepted"
- Idempotency key is written atomically in the same transaction as the match + points commitment
- `ChallengeIdempotency` records are small — acceptable for beta scale
- Production cleanup job may be needed at 2M-fan scale (Sprint 3 consideration)
